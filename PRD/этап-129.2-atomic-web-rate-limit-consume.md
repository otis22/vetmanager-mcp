# PRD: Этап 129.2 — atomic web rate-limit consume

## Контекст

Follow-up review 2026-04-21 выявил race в web auth limiter после stage 124:
- `check_rate_limit(...)` и `record_rate_limit_hit(...)` живут раздельно;
- register/login routes делают `check` и `record` в разных await-точках;
- при параллельных submit'ах несколько запросов могут одновременно увидеть count ниже лимита и пройти сверх configured threshold.

Проблема воспроизводится как в Redis path, так и концептуально в in-memory path, хотя в single-worker dev она менее заметна.

## Цель

Сделать web auth limiter атомарным на уровне consume semantics:
- один backend вызов должен проверять лимит и, если ещё можно, сразу записывать hit;
- login/register routes не должны собирать check+record вручную;
- lockout semantics остаются прежними: уже заблокированный ключ fast-fail'ится, а invalid login attempts учитываются атомарно.

## Scope

В scope:

## Подзадачи

### 129.2.1 Backend consume primitive

- добавить в backend protocol атомарный `consume_hit(...) -> (count_after, allowed)`
- `InMemoryRateLimitBackend`: реализовать consume без split check/record
- `RedisRateLimitBackend`: реализовать consume через optimistic transaction (`WATCH`/`MULTI`) или эквивалентный atomic path
- `_ResilientRedisBackend`: сохранить strict/fallback semantics и для consume

### 129.2.2 Web helper propagation

- добавить `consume_rate_limit(...)` в `web_security.py`
- оставить `check_rate_limit(...)`/`record_rate_limit_hit(...)` для test/backcompat, но auth routes больше не должны зависеть от split semantics

### 129.2.3 Auth route integration

- `/register`: заменить split check+record на consume path
- `/login`: сохранить fast-fail precheck, но invalid-credentials branch перевести на atomic consume; при переполнении отвечать `429`, не `401`
- success path по-прежнему чистит limiter state

### 129.2.4 Tests

- backend test на concurrent consume, где только `limit` запросов проходят
- web auth regression test на concurrent invalid logins: не больше `limit` ответов `401`, минимум один `429`
- сохранить существующие serial lockout tests зелёными

## Верификация

- `docker compose --profile test run --rm test pytest tests/test_rate_limit_backend.py tests/test_web_security.py tests/test_web_auth.py -q`
- `docker compose --profile test run --rm test`

## Риски

- Redis optimistic transaction может конфликтовать под высокой конкуренцией; нужен bounded retry loop.
- Login path использует два limiter key (`login`, `login_lockout`), между ними не будет cross-key atomicity; это допустимо, если каждый key-consume сам по себе atomic и route не пропускает over-limit request как `401`.
