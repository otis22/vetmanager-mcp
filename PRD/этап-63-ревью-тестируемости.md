# Этап 63. Ревью тестируемости (особенно E2E)

Цель: оценить покрытие, надёжность и поддерживаемость тестов; усилить E2E-контур.

---

## 63.1 Аудит покрытия

**Общие метрики:** 41 тестовый файл, ~8183 строк, 341 тест-функция, 101 MCP tool.

**Покрытие tools:** 98/101 (97%) — отличный показатель.

**3 непокрытых инструмента:**

| Tool | Модуль | Тип |
|---|---|---|
| `get_invoice_by_id` | tools/invoice.py | CRUD read |
| `get_medical_card_by_id` | tools/medical_card.py:141–152 | CRUD read |
| `update_medical_card` | tools/medical_card.py:200–230+ | CRUD write |

**Покрытие error scenarios — критический пробел:**

Протестировано только 2 HTTP-ошибки:
- 403 → AuthError
- 500 → VetmanagerError

Не протестировано:
- 400 Bad Request
- 401 Unauthorized
- 404 Not Found
- 422 Unprocessable Entity
- 429 Too Many Requests
- Timeout / connection errors
- Malformed JSON responses

## 63.2 Ревью E2E-тестов: стабильность

### Hardcoded waits — HIGH

`page.wait_for_timeout(50)` в двух browser-тестах:
- `test_browser_happy_path_user_token.py:39`
- `test_browser_real_opt_in.py:104`

50ms — хрупко при CI-нагрузке. Нужно заменить на `page.locator('[data-mode-panel="user_token"]').wait_for(state="visible")`.

### Fragile selectors — HIGH

`page.locator("h1").inner_text() == "Личный кабинет"` — generic h1 selector:
- `test_browser_happy_path_domain_api_key.py:31`
- `test_browser_happy_path_user_token.py:34`

Нужны data-testid атрибуты в HTML и соответствующие селекторы в тестах.

### Хорошие паттерны (сохранить):
- `page.wait_for_load_state("networkidle")` — корректно используется после навигаций
- Explicit cleanup через BrowserAccountCleanup с before/after отчётом

## 63.3 Ревью mock-контрактов

**Hardcoded response structures (124+ мест)** — каждый mock endpoint в test_e2e_mock.py возвращает захардкоженный JSON. Если API Vetmanager изменит envelope (например, `{"data": {"client": [...]}}` → другой формат), тесты не обнаружат расхождение.

**MedicalCards normalization** — tool обрабатывает оба варианта ключа ("medicalCards" и "medicalcards"), но mock тестирует только один путь.

**Рекомендация:** Добавить schema validation для mock-ответов на основе OpenAPI spec (`artifacts/vetmanager_openapi_v6.json`), чтобы моки не расходились с реальным API.

## 63.4 Test isolation

**Отличная изоляция** — проблем не найдено:
- `prepared_web_db` — fresh SQLite + env vars + global state reset на каждый тест
- `sqlite_session_factory_builder` — отдельный engine на тест с cleanup
- BrowserAccountCleanup — ORM cascades для удаления тестовых данных
- Все 341 тест не зависят от порядка выполнения
- Нет shared mutable state между тестами
