# Этап 129.5. Atomic Redis record_hit

## Контекст

После stage `129.2` atomic consume path уже есть, но `RedisRateLimitBackend.record_hit(...)`
по-прежнему делает `ZADD` и `EXPIRE` двумя отдельными await. При падении между ними ключ может
остаться без TTL.

## Цель

Сделать `record_hit(...)` атомарным Redis transaction path и убрать partial-write окно между
записью member и установкой TTL.

## Scope

1. Перевести `record_hit(...)` на transactional pipeline.
2. Переиспользовать общий helper для append+expire, чтобы логика не дублировалась с consume path.
3. Добавить regression test на то, что TTL выставляется вместе с записью.
