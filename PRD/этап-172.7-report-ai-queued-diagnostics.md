# Этап 172.7. Report AI queued diagnostics

## Контекст

Production feedback `#9`: пользователь видит, что Report AI job больше 30 секунд остаётся в `queued`, но MCP output не объясняет возраст очереди, не даёт operator hint и не оставляет отдельный low-cardinality signal для мониторинга таких poll-ов.

## Проверенные факты

- `get_report_ai_job` сейчас возвращает safe upstream job view через `GET /rest/api/report-ai-job/{id}` и не экспонирует raw SQL.
- Stage 170 уже требует bounded polling: агент должен честно показывать `queued/recognizing/building_preview`, не ждать бесконечно в одном call.
- `service_metrics.py` уже содержит process-local registry, `snapshot_service_metrics()` и `render_prometheus_metrics()` для Prometheus text output.
- `observability_logging.RUNTIME_LOGGER` — штатный runtime logger для structured event fields.
- MCP-side queued observation state and service metrics are process-local. In
  multi-worker deployment this signal is best-effort per process: polls routed to
  different workers can under-count observed age, and operators must use it as an
  MCP-side symptom, not as authoritative upstream queue duration.
- Vetmanager datetime convention in existing API artifacts is naive server-local
  `YYYY-MM-DD HH:MM:SS`; MCP process timezone is not guaranteed to match the
  Vetmanager server timezone.
- Report AI safe job view includes `created_at`, but this stage must not derive
  a 30-second queue age from naive upstream timestamps because timezone skew is
  larger than the threshold.
- Upstream-side worker/cleanup диагностика остаётся отдельным research gate; этот этап не чинит worker и не делает DB queries в upstream.

## Цель

Сделать long-queued Report AI polls видимыми и безопасно диагностируемыми на стороне MCP без PII/raw SQL: метрика + runtime log + safe output hint.

## Scope

### In scope

- Добавить low-cardinality metric counter для long-queued Report AI polls.
- Добавить safe runtime log event при long-queued poll.
- Дополнить output `get_report_ai_job` безопасным diagnostic block, если MCP наблюдает, что job находится в `queued` дольше threshold.
- Обновить описание `get_report_ai_job` и добавить operator runbook для stale in-progress диагностики.
- Regression tests на output, metric, log и отсутствие диагностики для fresh queued job.

### Out of scope

- Не менять upstream Report AI worker/cleanup.
- Не добавлять новые DB queries к upstream.
- Не логировать `intent_text`, recognized structure, candidates, raw SQL, клиентские данные или clinic domain.
- Не блокировать/не отменять queued job автоматически.

## Functional requirements

1. Long-queued threshold: 30 seconds.
2. `get_report_ai_job` должен считать queue age по MCP-observed duration:
   - при первом poll status exactly `queued` сохранить process-local monotonic timestamp для `job_id`;
   - при следующих queued polls считать `observed_queued_age_seconds = monotonic_now - first_seen_queued_monotonic`;
   - если status больше не `queued`, удалить локальное observation state для `job_id`;
   - observation key должен быть tenant-scoped: `(account_id, connection_id, job_id)`, без domain/secret в key/log/metric;
   - state должен быть bounded: max 4096 job entries, TTL eviction for entries not touched for 1 hour, cleanup on each access before insert/update, oldest-entry eviction if cap is still exceeded.
3. Upstream `created_at`/`updated_at` можно безопасно вернуть в diagnostics как raw safe fields, но нельзя использовать для 30-second threshold math.
4. Если status exactly `queued` и observed age >= 30 seconds, tool добавляет в `data.job.mcp_queue_diagnostics`:
   - `code="report_ai_job_long_queued"`;
   - `observed_queued_age_seconds` как целое число;
   - `threshold_seconds=30`;
   - `status="queued"`;
   - `created_at` из upstream, если есть;
   - `updated_at`, если есть;
   - `operator_hint` с безопасной инструкцией: продолжать bounded polling; если состояние сохраняется, проверить worker/stale in-progress jobs по runbook.
5. Metric increments once per long-queued poll, not per job lifetime. This is acceptable for poll observability.
6. Metric labels must be low-cardinality: no `job_id`, no domain, no intent.
7. Runtime log must include only safe fields: event name, status, threshold, age bucket and observed age seconds. `job_id` may be omitted to avoid cardinality/leak risk; if included, it must not be a metric label.
8. Fresh observed queued jobs (<30s) and non-queued jobs must not get diagnostics or metric increments.

## Acceptance criteria

- Test: a queued job observed by MCP for at least 30 monotonic seconds gets `mcp_queue_diagnostics` in `get_report_ai_job` output.
- Test: metric snapshot and Prometheus text expose the long-queued poll counter.
- Test: runtime warning log is emitted with safe event fields and without intent/raw SQL/domain.
- Test: fresh queued job does not get diagnostics and does not increment the counter.
- Test: two tenants with the same upstream `job_id` do not share observed queue age.
- Test: status transition away from `queued` clears the local observation state.
- Test: abandoned queued observations are evicted by TTL/cap cleanup and cannot grow without bound.
- Existing Report AI tests remain green.

## Decomposition

1. Observation state store: process-local bounded map with monotonic first-seen/last-seen timestamps, TTL cleanup, cap eviction, and transition cleanup.
2. Diagnostics annotation: add `mcp_queue_diagnostics` only for observed queued age >= 30 seconds; include only safe fields and operator hint.
3. Metrics/logging: add `record_report_ai_long_queued_poll()`, snapshot/Prometheus output, and safe runtime warning.
4. Runbook/docs: update `get_report_ai_job` tool description and add operator runbook with process-local/multi-worker caveat.
5. Tests: cover long-queued annotation, fresh queued no-op, transition cleanup, TTL/cap cleanup, metric, Prometheus text, and safe logging.

## Implementation notes

- Prefer small helpers in `tools/report_ai.py`:
  - bounded queued observation state;
  - monotonic observed age calculation;
  - diagnostics annotation.
- Use a module-level monotonic helper so tests can monkeypatch deterministic time.
- Add `record_report_ai_long_queued_poll()` to `service_metrics.py`, include it in reset/snapshot/Prometheus render.
- Keep operator runbook in `artifacts/report-ai-queued-diagnostics-runbook.md`.
