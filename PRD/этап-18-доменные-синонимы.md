# PRD: Этап 18. Применить справочник доменных имён и синонимов

## Цель

Использовать новый артефакт `artifacts/api_entity_reference-ru(с синонимами).md`
не как пассивную документацию, а как рабочий источник семантических подсказок для
MCP-клиентов и LLM. На практике это означает улучшить descriptions всех
зарегистрированных MCP-инструментов так, чтобы `tools/list` содержал доменные
синонимы сущностей Vetmanager и помогал модели точнее выбирать нужный инструмент.

## Контекст

- В проекте уже реализован контракт `tools/list`, и клиенты используют его как
  источник истины по возможностям сервера.
- Новый справочник содержит именно те пользовательские формулировки, которые
  должны улучшить tool selection: `владелец`, `запись на приём`, `приходная
  накладная`, `остаток на складе`, `прививка` и т.д.
- Текущие docstrings инструментов технически корректны, но местами слишком
  узкие с точки зрения доменного словаря.

## Решение

1. Создать централизованный каталог доменных синонимов для поддерживаемых
   сущностей проекта.
2. Поверх зарегистрированных FastMCP tools централизованно переопределить
   descriptions, чтобы не размазывать один и тот же словарь по десяткам модулей.
3. Добавить тесты на `tools/list`, проверяющие наличие enriched descriptions и
   representative synonyms у ключевых инструментов.
4. Зафиксировать контракт и архитектурное решение в README / AssumptionLog.

## Подзадачи

- 18.1 Создать каталог доменных синонимов и маппинг `tool -> entity`.
- 18.2 Применить enriched descriptions ко всем MCP-инструментам при регистрации.
- 18.3 Добавить тесты на `tools/list` для synonym-aware descriptions.
- 18.4 Обновить `README.md`, `Roadmap.md`, `AssumptionLog.md`.

## Границы

- В этом этапе не меняется бизнес-логика инструментов и не добавляются новые API
  вызовы.
- В этом этапе не затрагиваются prompts; фокус только на MCP tools и
  `tools/list`.
