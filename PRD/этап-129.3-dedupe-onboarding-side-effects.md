# PRD: Этап 129.3 — dedupe onboarding side effects

## Контекст

Post-review fix plan 2026-04-21 зафиксировал race в `save_user_login_password_connection(...)`:
- account-level save lock ставится только после `exchange_user_token(...)` и `validate_user_token_connection(...)`;
- два параллельных submit'а могут выпустить два разных user token во внешнем Vetmanager;
- в БД останется один active row, но внешний побочный эффект уже произошёл дважды.

## Цель

Не допускать duplicate upstream token issuance для одного `account_id` при параллельных onboarding submit'ах.

## Scope

В scope:
- `vetmanager_connection_service.py`
- `tests/test_vetmanager_connection_service.py`
- `Roadmap.md`, `AssumptionLog.md`

Вне scope:
- общий refactor всех save helpers
- cleanup Redis lifecycle
- deploy/release work

## Подход

- добавить per-account in-flight prepare registry для login/password onboarding;
- параллельные вызовы `save_user_login_password_connection(...)` должны await'ить один и тот же upstream exchange/validate task;
- перед insert проверить, не сохранил ли уже первый caller идентичную active connection; если да — вернуть её без disable/insert.

## Верификация

- новый concurrent test: два параллельных `save_user_login_password_connection(...)` делают ровно один `POST /token_auth.php`
- в БД остаётся одна active connection без лишней disabled duplicate row
- полный suite зелёный
