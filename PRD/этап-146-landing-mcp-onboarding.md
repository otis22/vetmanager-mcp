# PRD: Этап 146. Landing MCP onboarding for non-technical clinics

## Цель

Добавить на лендинг понятную секцию подключения ИИ-агента к Vetmanager через MCP для ветеринарных клиник и пользователей без технического опыта.

Секция должна объяснять пользу без жаргона, показывать клинические сценарии, давать copy-ready команды для Codex/Claude/Cursor/Manus и другого MCP-совместимого агента, а также подсказывать, что ключ доступа пользователь вставляет сам в настройки агента, а не отправляет в чат.

## Контекст

Лендинг реализован в `landing_page.py` через inline HTML/CSS. Предыдущие улучшения лендинга фиксировались в Stage 51, но текущая задача уже: добавить onboarding-блок для MCP-подключения.

Связанные проверенные факты из artifacts:

- Главный PRD фиксирует, что сервис предназначен для любых MCP-совместимых клиентов и должен помогать сотрудникам клиники работать с Vetmanager на естественном языке.
- Runtime-контракт актуальной архитектуры: MCP-клиенты используют `Authorization: Bearer <service_token>`, а credentials Vetmanager не передаются в аргументах инструментов.
- Web-контур уже включает landing/register/login/account; ключ доступа выдаётся пользователю через кабинет после настройки интеграции.
- Сервер публикует HTTP MCP endpoint по пути `/mcp`; production base URL управляется через `SITE_BASE_URL` с fallback на `https://vetmanager-mcp.vromanichev.ru`.

Аудитория:

- владельцы и руководители ветеринарных клиник;
- администраторы и врачи, далекие от программирования;
- пользователи готовы работать с Codex/Claude/Cursor/Manus, если им дать простую инструкцию;
- Codex позиционируется как рекомендуемый старт, но Vetmanager MCP должен выглядеть универсальным для любых MCP-совместимых агентов.

## Проблема

Текущий термин “MCP” непонятен нетехнической аудитории. Если сразу показывать конфиги, `Bearer token`, JSON или системные промпты, пользователь не понимает ценность и боится сломать настройку.

Нужно сначала показать пользу для клиники, затем объяснить MCP как мост, затем дать простую команду агенту: “настрой MCP, путь к файлу покажи, ключ доступа я вставлю сам”.

## In Scope

1. Добавить на лендинг секцию “Подключите ИИ-агента к вашему Vetmanager за 5 минут”.
2. Объяснить MCP как “мост между ИИ-агентом и Vetmanager”, без protocol details.

8. Добавить fallback “Если агент не открыл файл настроек” с ручными подсказками.

## Out of Scope

- Не добавлять системные промпты.
- Не просить пользователя вставлять ключ доступа или Bearer token в чат.
- Не показывать JSON/config как основной сценарий.
- Не объяснять JSON-RPC, stdio, HTTP transport, protocol schema.
- Не добавлять видео как основной формат инструкции.
- Не добавлять список всех возможных MCP-агентов сверх Codex/Claude/Cursor/Manus/Другой агент.
- Не менять серверный MCP contract.
- Не менять регистрацию, логин, кабинет аккаунта и token issuance flow, кроме ссылок/текста, если они уже используются лендингом.

## UX / Content Requirements

### Главный заголовок секции

```text
Подключите ИИ-агента к вашему Vetmanager за 5 минут
```

### Подзаголовок

```text
Работает через MCP: Codex, Claude, Cursor, Manus и другие совместимые агенты смогут находить клиентов, смотреть записи, проверять счета и считать выручку по данным вашей клиники.
```

### Простое объяснение MCP

```text
MCP — это мост между ИИ-агентом и Vetmanager. Вы задаёте вопрос обычным языком, агент обращается к Vetmanager через разрешённые команды и возвращает ответ по вашим данным.
```

Запрещённые слова в основном тексте секции:

- `JSON-RPC`;
- `stdio`;
- `transport`;
- `schema`;
- `endpoint`;
- “протокол” как центральное объяснение.

Допустимо использовать `MCP` и `Bearer token`, но `Bearer token` должен идти как уточнение к “ключу доступа”, а не как основной термин.

### Визуальная схема

Схема должна читаться как пользовательский сценарий:

```text
Вы задаёте вопрос
↓
ИИ-агент
↓
MCP-мост
↓
Ваш Vetmanager
↓
Ответ по данным клиники
```

На desktop схема может быть горизонтальной, на mobile — вертикальной. Подписи должны объяснять, что идёт по стрелкам.

### Payoff-блок до инструкции

Добавить блок “Что можно спросить после подключения” до технических шагов:

- “Какая выручка была за март?”
- “Покажи записи врача на завтра”
- “Найди клиента по телефону”
- “Какие счета оплачены частично?”
- “Кому из пациентов пора на прививку?”

Эти фразы должны выглядеть как готовые вопросы, а не как список API-возможностей.

### Подключение в 3 шага

```text
1. Выберите агента
2. Скопируйте и отправьте ему команду настройки
3. Вставьте ключ доступа в настройки и перезапустите сессию
```

Добавить заметку:

```text
Настройку удобнее делать с компьютера.
```

### Privacy note

```text
Ключ доступа не нужно отправлять в чат. Он хранится в настройках агента или на вашем компьютере.
```

Если нужен термин `Bearer token`, использовать:

```text
ключ доступа / Bearer token
```

## Команды Для Агентов

Команды должны быть доступны для копирования. Плейсхолдер адреса сервера:

```text
<MCP_SERVER_URL>
```

В реализации плейсхолдер заменяется на реальный URL:

```text
{resolved_SITE_BASE_URL}/mcp
```

Для production default это:

```text
https://vetmanager-mcp.vromanichev.ru/mcp
```

Если self-hosted оператор задаёт `SITE_BASE_URL`, команды должны использовать этот base URL. В финальном HTML не должно оставаться `<MCP_SERVER_URL>`.

Техническое решение для `landing_page.py`: вычислить `base_url = _resolve_site_base_url()` и `mcp_url = f"{base_url}/mcp"` в `render_landing_page()`, затем подставить `mcp_url` через отдельный HTML placeholder, не похожий на HTML tag, например `__MCP_SERVER_URL__`. Не использовать `str.format()` для всей HTML-строки, потому что CSS содержит фигурные скобки. Перед `return html` добавить defensive guard:

```python
if "__MCP_SERVER_URL__" in html:
    raise RuntimeError("MCP server URL placeholder was not replaced")
```

Тесты должны проверять default URL, self-hosted URL через `SITE_BASE_URL` и отсутствие placeholder в итоговом HTML.

Новая onboarding-секция не должна встраивать production default URL напрямую. Внутри неё использовать только `__MCP_SERVER_URL__`, который заменяется на вычисленный `mcp_url`. Existing глобальный `html.replace(_DEFAULT_SITE_BASE_URL, base_url)` можно оставить для уже существующих canonical/OG/technical snippets, но тесты Stage 146 должны проверять, что при override `SITE_BASE_URL` внутри `mcp-onboarding` нет stray `https://vetmanager-mcp.vromanichev.ru/mcp`.

## Где Взять Ключ Доступа

Добавить CTA рядом с инструкциями:

```text
Ключ доступа выдаётся в кабинете после регистрации и подключения Vetmanager.
```

Кнопки/ссылки:

- `Создать аккаунт` → `/register`
- `Войти в кабинет` → `/login` (пользователь после входа попадает в account flow; прямую ссылку на `/account` не показывать как основной CTA, чтобы не вести неавторизованного пользователя на protected route)

Для пользователей, которые уже вошли:

```text
Откройте кабинет, проверьте подключение Vetmanager и создайте service token. Если ключ уже был создан, можно выпустить новый и заменить его в настройках агента.
```

Это статический текст для всех посетителей лендинга, без session-aware rendering в `render_landing_page()`. Лендинг не должен определять auth state; CTA остаются `/register` и `/login`.

## Fallback: Если Агент Не Открыл Файл Настроек

Добавить видимый блок рядом с командами, не только в FAQ:

```text
Если агент не открыл файл настроек, попросите его ещё раз показать путь к конфигурации. Если не получилось, откройте ручную инструкцию для вашего агента.
```

Минимальный список ручных подсказок:

- Codex: файл/настройки Codex MCP config; попросить Codex показать точный путь в вашей системе.
- Claude: настройки Claude Desktop / MCP servers; после изменения перезапустить Claude.
- Cursor: MCP settings в Cursor; после изменения перезапустить Cursor-сессию.
- Manus: настройки подключений/инструментов Manus; проверить список доступных tools.
- Другой агент: найти раздел MCP servers, указать URL и авторизацию через ключ доступа.

Если точные OS-specific пути неизвестны или различаются, не выдумывать их в UI. Вместо этого давать инструкцию “попросите агента показать путь для вашей системы”.

## Interaction Contract

Для вкладок и копирования допустим минимальный inline JavaScript в `landing_page.py`, потому что существующий лендинг является static inline HTML и copy UX требует состояния.

Требования:

- Вкладки реализовать как semantic buttons с `role="tab"`, `aria-selected`, `aria-controls`; панели — с `role="tabpanel"`.
- No-JS fallback: все пять tabpanel должны быть server-rendered и читаемы в HTML. JavaScript при загрузке усиливает секцию: оставляет видимой активную Codex-панель, скрывает остальные через `hidden`, и переключает панели по клику.
- Codex активен по умолчанию: ровно один tab имеет `aria-selected="true"` на initial render.
- Кнопки копирования должны иметь `data-copy-target="<pre-id>"`, доступное имя, `aria-live="polite"` и состояние “Скопировано”.
- При переключении вкладки показывается соответствующая команда; JS обновляет `aria-selected`, `tabindex` и `hidden` у tabpanels.
- Если Clipboard API недоступен или `writeText` завершился ошибкой, кнопка остаётся в состоянии “Скопировать”, ошибка не пробрасывается, текст команды остаётся видимым для ручного копирования.
- При успешном копировании label кнопки становится “Скопировано” на 2000 ms, затем возвращается в “Скопировать”.
- Использовать стабильный контейнер секции: `id="mcp-onboarding"` и `data-testid="mcp-onboarding"` для тестов.
- Основной marketing-текст внутри секции пометить контейнером `data-testid="mcp-onboarding-main-copy"` для проверки отсутствия технического жаргона.
- В `mcp-onboarding-main-copy` должны входить: заголовок секции, подзаголовок, простое объяснение MCP, подписи визуальной схемы, payoff-вопросы, 3 шага подключения, privacy note и CTA о получении ключа доступа.
- В `mcp-onboarding-main-copy` не входят: command `<pre>` blocks, fallback details с конфигурационными подсказками, common-errors body и role-example cards.
- Copy contract: каждая кнопка копирования имеет `data-copy-target="<pre-id>"`, а JS читает `textContent` из соответствующего `<pre>`. Видимый `<pre>` является единственным источником текста команды. Не хранить полный multi-line command в HTML attribute.

## Частые Ошибки

Блок должен быть видимым, с формулировками по симптомам пользователя:

1. Агент не видит Vetmanager.
   - Перезапустите сессию агента.
   - Попросите агента показать список подключённых MCP-серверов.
2. Ошибка 401 / ключ доступа не подошёл.
   - Проверьте, что ключ вставлен в настройки, а не в чат.
   - Проверьте, что ключ скопирован полностью.
3. Не знаю, куда вставить ключ.
   - Попросите агента показать путь к файлу или настройке.
   - Откройте ручную подсказку для вашего агента.
4. Настроил, но инструменты не появились.
   - Перезапустите сессию или приложение.
   - Проверьте адрес MCP-сервера.

## Визуальные Требования

- Секция должна быть светлой, спокойной, без developer-console эстетики.
- Не использовать терминальный экран как главный визуал.
- Не делать code/config первым элементом секции.
- Вкладки должны быть понятными и не перегруженными.
- Кнопка “Скопировать” должна иметь подтверждение состояния: “Скопировано”.
- На mobile вкладки/команды не должны ломать ширину экрана; длинные строки должны переноситься.
- Не использовать hero-scale typography внутри compact panels.
- Не использовать декоративные gradient blobs/orbs.
- Секция должна выглядеть как мастер подключения, а не как документация API.

## Acceptance Criteria

- На лендинге есть секция с заголовком “Подключите ИИ-агента к вашему Vetmanager за 5 минут”.
- В основном тексте MCP объясняется как мост между агентом и Vetmanager.
- Секция имеет `id="mcp-onboarding"` и `data-testid="mcp-onboarding"`.
- До инструкций есть payoff-блок с минимум 5 клиническими вопросами.
- Есть 3 шага подключения.
- Есть вкладки/переключатели Codex, Claude, Cursor, Manus, Другой агент.
- Codex помечен как “Рекомендуем для старта”.
- Каждая вкладка содержит copy-ready команду с реальным MCP URL `{SITE_BASE_URL}/mcp`; в HTML не остаётся `<MCP_SERVER_URL>`.
- Каждая команда содержит фразу о том, что пользователь сам вставит ключ доступа / Bearer token, а агент должен показать путь к файлу или настройке.
- В основном UI есть privacy note: ключ доступа не отправляется в чат.
- Есть CTA/ссылка, где получить ключ доступа: регистрация и вход в кабинет.
- Текст про уже зарегистрированных пользователей является статическим и не требует auth-state detection.
- Есть fallback “Если агент не открыл файл настроек”.
- Есть примеры задач по ролям: администратор, врач, руководитель клиники.
- Есть видимый блок “Частые ошибки”.
- В основном marketing-тексте секции (`data-testid="mcp-onboarding-main-copy"`) нет `JSON-RPC`, `stdio`, `transport`, `schema`, `endpoint`.
- В секции нет инструкции вставлять ключ доступа или Bearer token в чат.
- Вкладки имеют semantic ARIA contract (`role="tablist"`, `role="tab"`, `role="tabpanel"`).
- На initial render все tabpanels присутствуют в HTML; JS содержит enhancement, который скрывает неактивные панели после загрузки.
