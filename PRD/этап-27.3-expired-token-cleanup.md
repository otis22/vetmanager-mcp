# PRD: Этап 27.3. Cleanup/revocation policy для истёкших токенов

## Цель

Зафиксировать и реализовать предсказуемую политику для Bearer-токенов, срок
действия которых истёк, чтобы они:
- больше не выглядели активными в кабинете;
- не требовали обязательного runtime-запроса для перехода в terminal state;
- оставляли audit след при cleanup.

## Текущее поведение

- `ServiceBearerToken.is_expired()` и `sync_status()` уже умеют вычислить
  состояние `expired`.
- Но токен может оставаться сохранённым в БД со статусом `active`, пока его:
  - не проверит runtime auth path;
  - или явно не тронет код, который вызывает `sync_status()`.
- Из-за этого web-кабинет может показывать просроченный токен как `active`,
  если с ним давно не было runtime activity.

## Выбранная политика

- Истёкший токен не автоконвертируется в `revoked`.
- Cleanup sweep переводит его в `expired`.
- Cleanup дополнительно пишет audit event `token_expired`.
- Revoked token остаётся `revoked` и не переопределяется cleanup-логикой.

## Почему не auto-revoke

- `expired` и `revoked` — разные причины неактивности.
- `revoked` отражает явное управленческое действие пользователя или сервиса.
- `expired` отражает истечение TTL и полезен как отдельный эксплуатационный
  статус для UI и аудита.

## Точки интеграции

- Новый cleanup helper/service:
  - ищет активные токены с `expires_at <= now`;
  - переводит их в `expired`;
  - пишет `token_expired`.
- Web dashboard:
  - перед построением списка токенов выполняет cleanup sweep для аккаунта.

## Что не входит в этот шаг

- Физическое удаление старых токенов из БД.
- Периодический background scheduler.
- Глобальная cron-like cleanup orchestration вне request path.

## Декомпозиция

### 27.3.1 Cleanup helper
- Добавить функцию синхронизации expired tokens по аккаунту или глобально.
- Гарантировать, что revoked tokens не затрагиваются.

### 27.3.2 Audit integration
- При первом cleanup expired token писать `token_expired`.
- Не создавать дубликаты event при повторном sweep после уже переведённого статуса.

### 27.3.3 Web integration
- Встроить cleanup sweep в загрузку кабинета аккаунта.

### 27.3.4 Tests
- Unit test на перевод `active -> expired`.
- Unit test на отсутствие повторных event при повторном sweep.
- Web test на отображение expired token в кабинете без runtime resolve.

## Критерии готовности

- Просроченные токены перестают отображаться как `active` в кабинете.
- Cleanup пишет `token_expired` один раз на токен.
- Revoked tokens не переопределяются cleanup sweep.
