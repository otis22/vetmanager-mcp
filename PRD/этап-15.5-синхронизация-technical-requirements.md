# PRD Этап 15.5: Синхронизация technical requirements с текущей архитектурой

## Цель
Обновить `artifacts/technical-requirements-vetmanager-mcp-ru.md`, чтобы он
соответствовал текущей реализации проекта и больше не вводил в заблуждение
при планировании следующих этапов.

## Проблема
- Артефакт всё ещё описывает env-based credentials через
  `VETMANAGER_DOMAIN` / `VETMANAGER_API_KEY`.
- В документе фигурируют устаревшие элементы структуры проекта:
  `.venv`, `config.py`, `requirements.txt`, запуск через `uv`.
- Указан старый transport (`stdio`) вместо текущего HTTP MCP сервера.
- Документ не отражает headers-only контракт, мультитенантность,
  кеширование, pacing запросов и security hardening, уже реализованные в коде.

## Границы задачи
- Обновить только архитектурный артефакт, `Roadmap.md` и `AssumptionLog.md`.
- Не менять код сервера, tools или docker-конфигурацию, если не найдена
  фактическая ошибка реализации.

## Декомпозиция

### 15.5.1 Обновление содержания technical requirements
- Переписать разделы про стек, архитектуру, структуру проекта, аутентификацию,
  запуск, тестирование и деплой под текущую реализацию.
- Зафиксировать:
  - Python 3.11+ / фактическую сборку на Python 3.12 slim;
  - FastMCP + `streamable-http`;
  - headers-only credentials;
  - docker-only workflow;
  - in-memory tagged cache;
  - pacing/retry/timeout/security ограничения.

### 15.5.2 Согласованность артефактов
- Сверить обновлённый документ с:
  - `README.md`;
  - `Roadmap.md`;
  - `artifacts/prd-vetmanager-mcp-ru.md`.
- Не допускать противоречий по runtime credentials, transport и способу запуска.

### 15.5.3 Фиксация решения
- Обновить статус Stage 15.5 в `Roadmap.md`.
- Добавить запись в `AssumptionLog.md`.

## Критерии готовности
- `technical-requirements-vetmanager-mcp-ru.md` больше не содержит устаревшую
  env-based архитектуру и старую структуру проекта.
- Документ согласован с README, главным PRD и текущим кодом.
- Решение зафиксировано в `AssumptionLog.md`.
