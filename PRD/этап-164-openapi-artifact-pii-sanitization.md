# Этап 164. OpenAPI artifact PII and credential-derived examples sanitization

## Контекст

Security/privacy audit 2026-05-17 обнаружил в `artifacts/vetmanager_openapi_v6.json` concrete email examples и `passwd` hash-like examples. Это противоречит `artifacts/prd-vetmanager-mcp-ru.md` §4.2.11-4.2.12: Vetmanager credentials и тестовые значения не должны попадать в проектные артефакты как concrete secrets/PII.

Stage 163 закрыл historical API-key-like literal в `AssumptionLog.md`. Stage 164 не прячет и не удаляет OpenAPI artifact; меняются только значения примеров, не структура API contract.

## Цель

Сохранить `artifacts/vetmanager_openapi_v6.json` как рабочий reference source, но заменить concrete emails и credential-derived `passwd` examples на нейтральные placeholders без изменения schema/field names/types.

## Scope

- Санитизация только concrete PII/credential-derived values в reference artifacts.
- Основной файл: `artifacts/vetmanager_openapi_v6.json`.
- Neighbor audit: `artifacts/vetmanager_postman_collection.json`, `artifacts/api_entity_reference-ru.md`, `artifacts/api-research-notes-ru.md`.
- Lightweight regression script/check, который ловит возврат известных concrete email/hash literals.
- Contract-preservation script/check: `scripts/check_openapi_artifact_contract_preserved.py`.
- Audit artifact: `artifacts/security/stage-164-openapi-privacy-audit.md` со списком JSON paths, value classes, SHA-256 fingerprints и decisions без raw concrete literals.
- Baseline artifact: `artifacts/security/stage-164-openapi-structure-baseline.json` со structural fingerprint metadata pre-sanitization OpenAPI без raw values.

## Out of scope

- Удаление или скрытие OpenAPI artifact.
- Git history rewrite для уже раскрытых значений; current-tree sanitization не удаляет значения из истории, чужих клонов, forks/caches.
- Обновление runtime code/tools.
- General cleanup generated artifacts.
- Sanitization `.playwright-mcp` snapshots: нет подтверждённого High/Critical PII/secret finding в Stage 164; если Stage 165 inventory подтвердит риск, добавить отдельный narrow stage.
- Широкий generic secret scanner для всего repo: Stage 163 уже добавил exact historical check; Stage 165 отвечает за full critical findings inventory.

## Placeholder policy

- Concrete personal/vendor emails заменяются на reserved RFC 2606/6761-style domains:
  - `user@example.com`
  - `clinic-user@example.com`
  - `generated-user@example.test`
- Mapping deterministic: same original literal always maps to the same placeholder. Classification heuristic:
  - `clinic-user@example.com`: local-part or domain is tied to clinic/vendor/operator context by surrounding OpenAPI example fields or known product/vendor domain.
  - `generated-user@example.test`: local-part/domain shape is clearly autogenerated or disposable/random-looking.
  - `user@example.com`: generic mailbox examples that are not tied to clinic/vendor/operator context and not clearly autogenerated.
- Audit artifact records the SHA-256 fingerprint, classification and triggering classification criterion for every sanitized email literal without recording raw value.
- `passwd` hash-like examples заменяются на neutral fixed placeholder hash `00000000000000000000000000000000`, чтобы сохранить JSON type/shape и не имитировать реальный credential.
- Перед заменой `passwd` проверяется фактический schema context: затронутые поля имеют string value, `x-db-type: varchar(32)` и не имеют OpenAPI `pattern`/`format`/`minLength`/`maxLength`; placeholder обязан сохранять длину 32. Если затронутый `passwd` occurrence нарушает precondition, Stage 164 abort'ит sanitization для этого field, фиксирует `context-mismatch` в audit artifact с JSON path/fingerprint без raw value и добавляет отдельный narrow Roadmap follow-up.
- Field names (`email`, `passwd`) и schema structure не меняются.

## Acceptance criteria

- `artifacts/vetmanager_openapi_v6.json` остаётся валидным JSON.
- В `artifacts/vetmanager_openapi_v6.json` не остаётся известных concrete values:
  - real-looking emails from the 2026-05-17 audit;
  - repeated `passwd` hash-like example from the 2026-05-17 audit.
- Neighbor audit по Postman/API docs выполнен и не расширяет scope без concrete matches.
- Если neighbor audit находит concrete PII/credential-derived values того же класса, Stage 164 санитизирует их по той же placeholder policy, добавляет их в audit artifact и regression check; если finding требует другой политики, он фиксируется отдельным Roadmap stage.
- `artifacts/security/stage-164-openapi-privacy-audit.md` содержит JSON paths, value classes, SHA-256 fingerprints, placeholder classification criteria, schema-context decisions и neighbor audit result без raw concrete literals.
- Добавлен deterministic regression check, который:
  - называется `scripts/check_reference_artifact_privacy.py`;
  - сканирует union of Stage 164 reference artifacts: `artifacts/vetmanager_openapi_v6.json`, `artifacts/vetmanager_postman_collection.json`, `artifacts/api_entity_reference-ru.md`, `artifacts/api-research-notes-ru.md`;
  - fail'ит только по deny-list fingerprint'ам известных concrete literals из аудита 2026-05-17;
  - не хранит и не печатает raw concrete literals;
  - не считает `example.com`, `example.test` и neutral `passwd` placeholder нарушениями.
- Fingerprint algorithm: SHA-256 over UTF-8 bytes. Email candidates are normalized by trimming surrounding whitespace and lowercasing the full email before hashing; `passwd` hash candidates are trimmed but otherwise exact ASCII strings. Encoding errors are fatal.
- Проверка contract preservation сравнивает structural fingerprint текущего OpenAPI с committed baseline file `artifacts/security/stage-164-openapi-structure-baseline.json`, который создаётся до sanitization из текущего dirty-free OpenAPI, коммитится в этом же stage и не регенерируется runtime'ом. Refresh baseline разрешён только отдельным осознанным artifact-contract stage. Baseline содержит hash/count metadata без raw values: JSON object keys, array lengths, scalar value types и affected `passwd` schema context должны совпасть.
- Baseline file generated in Stage 164 from pre-sanitization `HEAD:artifacts/vetmanager_openapi_v6.json`; audit artifact records the source git SHA and pre-sanitization structural fingerprint metadata used to derive it, so baseline origin is independently verifiable.
- `scripts/check_openapi_artifact_contract_preserved.py` exists, reads only `artifacts/vetmanager_openapi_v6.json` and `artifacts/security/stage-164-openapi-structure-baseline.json`, verifies JSON object keys, array lengths, scalar value types and affected `passwd` schema context (`type: string`, `x-db-type: varchar(32)`, no standard OpenAPI pattern/format/min/max constraints, placeholder length 32), and is the pass/fail contract-preservation gate.
- AssumptionLog фиксирует:
  - artifact не скрывался и fixed security note остаётся видимым;
  - current-tree sanitization не удаляет git-history exposure;
  - git-history rewrite намеренно out-of-scope;
  - происхождение значений не подтверждено как synthetic/expired/non-production, поэтому external mitigation для реального PII/credential exposure остаётся operator responsibility или отдельным narrow follow-up, если пользователь решит её выполнять.
- `git diff --check` и full suite проходят.

## Проверки

Red:

```bash
python3 scripts/check_reference_artifact_privacy.py
```

Green:

```bash
python3 scripts/check_reference_artifact_privacy.py
python3 scripts/check_openapi_artifact_contract_preserved.py
python3 -m json.tool artifacts/vetmanager_openapi_v6.json >/dev/null
git diff --check
docker compose --profile test run --rm test
```

`check_reference_artifact_privacy.py` is the pass/fail privacy gate for all Stage 164 reference artifacts. Optional broad `rg` output is triage-only because reserved placeholders intentionally look like emails and the neutral `passwd` placeholder intentionally preserves hash-like shape.

`check_openapi_artifact_contract_preserved.py` reads only `artifacts/vetmanager_openapi_v6.json` and `artifacts/security/stage-164-openapi-structure-baseline.json`. It is intentionally structural, not semantic OpenAPI validation: the artifact uses vendor metadata such as `x-db-type`, and the affected `passwd` fields do not define standard OpenAPI pattern/format/min/max constraints. The script must verify the available constraints relevant to this replacement, especially string type and 32-character `varchar(32)` shape.

## Simplicity decision

Самое простое корректное решение: deterministic replacement of known concrete examples plus a narrow regression script. Не добавляем большой scanner/cleanup framework в Stage 164; это снизит риск drift в OpenAPI artifact и не превратит задачу в неограниченный cleanup.
