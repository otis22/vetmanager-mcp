# Этап 172.4. Report AI goods `good.id` workaround

## Контекст

Production feedback `#7`: Report AI построил отчёт по товарам, но preview упал с `PREVIEW_FAILED` из-за SQL с недоступной колонкой/выражением `good.id`. Upstream `PromptBuilder` уже содержит hints про корректные связи (`invoice_document.good_id = good.id`), но production кейс всё равно показал ошибку. Корневой фикс относится к upstream Report AI schema/prompt mapping.

## Проверенные факты

- Upstream `ReportAiJob::toSafeArray()` возвращает `status`, `recognized` без raw SQL, `preview_summary`, `report_id`, `error_code`, `error_message_safe`, `created_at`, `updated_at`, `saved_at`.
- Upstream `JobService::runJob()` при build error вызывает `markFailed($jobId, PREVIEW_FAILED, safeMessage(...))`; safe message обрезается до 500 символов, но не гарантирует наличие raw SQL.
- `get_report_ai_job` возвращает safe job body и не поднимает `ToolError` для terminal `failed` status.
- `get_report_ai_job_data` при upstream `PREVIEW_FAILED` получает HTTP 500 и в MCP становится `ToolError`, поэтому именно этот path может использовать generic known-issue injection.

## Цель

Добавить MCP-level deterministic workaround, чтобы агент не повторял проблемную формулировку и получал безопасную подсказку при уже упавшей job.

## Scope

### In scope

- Обновить `report_ai_prompt_helper`: для товарных отчётов просить код/артикул/наименование товара, а не отдельную колонку `good.id`.
- В `get_report_ai_job` добавить safe annotation, когда upstream job вернулась как `failed` + `PREVIEW_FAILED` и safe error text содержит `good.id`/unknown-column markers.
- Добавить seed `known_issues` row для `get_report_ai_job_data`, чтобы real ToolError path тоже получал deterministic playbook.
- Покрыть prompt, payload annotation и seed matcher тестами.

### Out of scope

- Не менять upstream `vetmanager-extjs`.
- Не пытаться переписывать SQL или raw recognized payload на MCP стороне.
- Не раскрывать raw SQL в MCP output.

## Дизайн

### Prompt helper

Добавить в helper раздел про goods reports:

- если нужен товарный отчёт, формулировать требуемые поля как "код/артикул/наименование товара";
- не просить отдельную колонку `good.id`;
- для продаж товаров указывать связь через позиции счёта и товары на уровне бизнес-терминов, без SQL.

### Payload annotation

`get_report_ai_job` возвращает upstream safe payload. Если `job.status == "failed"`, `job.error_code == "PREVIEW_FAILED"` и `job.error_message_safe` содержит маркеры `good.id`, `unknown column`, `unknown field` или русские аналоги, MCP добавляет:

```json
{
  "mcp_workaround": {
    "code": "report_ai_goods_good_id_preview_failed",
    "summary": "...",
    "steps": [...],
    "safe_to_retry": true
  }
}
```

Annotation не меняет upstream статус и не скрывает ошибку.

Если safe text не содержит goods markers, annotation не добавляется. MCP не должен пытаться угадать проблему по одному `PREVIEW_FAILED`, потому что этот error code покрывает и другие причины.

### Known issue seed

Seed issue должен match-иться на `related_tool=get_report_ai_job_data`, `error_code=ToolError`, `normalized_error_text` с `preview_failed` и отдельным goods marker (`good.id`, quoted `good`.`id`, `good id`, русские варианты). Одного `preview_failed` недостаточно.

Playbook должен рекомендовать переформулировать `intent_text` через `report_ai_prompt_helper` и заново создать job.

## Acceptance

- Prompt helper содержит явный goods workaround.
- `get_report_ai_job` добавляет `mcp_workaround` только для целевого failed/PREVIEW_FAILED goods marker case.
- Seed validation проходит, seeded issue match возвращает agent playbook.
- Targeted Docker tests проходят.
