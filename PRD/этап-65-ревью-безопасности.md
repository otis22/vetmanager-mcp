# Этап 65. Ревью безопасности

Цель: комплексный аудит безопасности — аутентификация, хранение секретов, инъекции, заголовки, конфигурация.

---

## 65.1 Аудит auth chain

### CRITICAL: Разные сообщения об ошибках auth раскрывают состояние токена

**bearer_auth.py:87, 103, 118, 121** — разные сообщения:
- "Invalid bearer token." (не найден)
- "Revoked bearer token." (отозван)
- "Expired bearer token." (истёк)

Атакующий может определить, какие токены существуют, отозваны или истекли.

**Fix:** Единое сообщение `"Invalid authorization."` для всех случаев. Детали оставить в audit логах.

### HIGH: Timing attack в verify_account_password

**web_auth.py:70–80** — early returns (строки 70–71, 74–75, 80) пропускают PBKDF2 вычисление:
- `if not password_hash: return False` — нет хеша (аккаунт не существует)
- `if scheme != PASSWORD_SCHEME: return False` — невалидный формат
- `except: return False` — ошибка парсинга

Разница во времени ответа позволяет отличить "аккаунт не существует" от "неправильный пароль".

**Fix:** Всегда выполнять PBKDF2 с dummy salt при ранних выходах.

### MEDIUM: Session fixation

**web.py:1092, 1185** — session cookie устанавливается после auth без инвалидации предыдущей.

**Fix:** Очищать старый cookie перед установкой нового.

### MEDIUM: Session без random nonce

**web_auth.py:120–131** — session token = `account_id.timestamp.signature`. Timestamp с точностью до секунды, account_id предсказуем.

**Fix:** Добавить random nonce: `account_id.timestamp.nonce.signature`.

## 65.2 Аудит хранения секретов

### VERIFIED: .env не в git

`.env` в `.gitignore`, не отслеживается в git history. Секреты только в локальном файле — OK.

### Encryption: AES через Fernet — OK

`secret_manager.py` использует cryptography.fernet.Fernet — AES-128-CBC с HMAC. Корректная реализация.

### MEDIUM: Нет валидации формата ключа при startup

`get_storage_encryption_key()` проверяет только что ключ не пустой, но не валидирует что это валидный Fernet key.

**Fix:** Попытка создать `Fernet(key)` при startup.

## 65.3 Аудит input validation

### Filter/sort параметры

`validators.py` валидирует `limit` (1–100) и `offset` (0–10000). Filter и sort принимаются как JSON string и передаются в Vetmanager API as-is.

**Risk:** LOW — параметры идут в query string к upstream API, не в SQL. Vetmanager API сам валидирует. Injection risk минимален, т.к. нет interpolation в наш код.

### Tool arguments

Все tool-аргументы типизированы (int, str, Optional). Нет concatenation в SQL или shell commands. Risk: LOW.

## 65.4 Аудит HTTP-заголовков

### CSP: style-src 'unsafe-inline' — MEDIUM (known, TD-55-02)

Уже зафиксировано в tech debt register. Запланировано в этапе 58.

### Остальные заголовки — OK

- `X-Frame-Options: DENY` — OK
- `X-Content-Type-Options: nosniff` — OK
- `Referrer-Policy: no-referrer` — OK
- `Content-Security-Policy` с script nonce — OK
- Cookie: `httponly=True, secure=True (prod), samesite=Lax` — OK

## 65.5 Аудит конфигурации

### Server binding на 0.0.0.0 — LOW

`server.py` слушает `0.0.0.0` по умолчанию. В Docker это нормально (проксируется через nginx). В bare-metal — нужна настройка.

### Debug mode — OK

Нет debug flag, нет verbose error output в production responses.

### Error messages — OK

Ошибки клиенту не содержат stack traces. Sentry ловит unhandled exceptions с redaction.

### Что НЕ является проблемой (false positives из аудита):

- **.env в git** — FALSE. `.env` в `.gitignore`, не отслеживается.
- **TLS certificate pinning** — UNNECESSARY для SaaS API. Vetmanager API использует стандартные CA, pinning создаст operational burden без реальной выгоды.
