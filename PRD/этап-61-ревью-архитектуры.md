# Этап 61. Ревью архитектуры (code smells)

Цель: выявить архитектурные запахи, нарушения SRP, скрытые связности, дублирование, god-objects и составить приоритизированный план устранения.

---

## 61.1 Ревью server.py, vetmanager_client.py, web.py

### server.py (49 строк) — ЧИСТО

Минимальный bootstrapping-модуль. Каждый шаг делегирован специализированному модулю. Нарушений SRP нет.

### vetmanager_client.py (304 строки) — HIGH severity

**God-object: 8+ ответственностей в одном классе.**

| Ответственность | Строки | Суть |
|---|---|---|
| HTTP-запросы | GET/POST/PUT/DELETE обёртки | Основная задача |
| Аутентификация | 81–102 | Резолв credentials из bearer token |
| Кеширование | 104–149, 253–260 | Cache key, TTL, инвалидация |
| Rate limiting | 138–145, 77–78 | Пауза между запросами к upstream |
| Host resolution | 150–205 | Billing API → base URL |
| Scope checking | 212–220 | Проверка прав токена |
| Error translation | 277–292 | HTTP status → domain exceptions |
| Observability | метрики, логи | record_upstream_failure, RUNTIME_LOGGER |

**Конструктор инициализирует 8 разных concern-ов:**
```python
self._vetmanager_auth = None       # Auth context
self._domain = None                # Domain state
self._api_key = None               # Credentials
self._base_url = None              # Host resolution cache
self._last_request_started_at = 0.0 # Pacing state
self._pace_lock = asyncio.Lock()   # Pacing lock
self._credentials_lock = asyncio.Lock()  # Credential lock
```

**Скрытые связности:** request_cache, service_metrics, runtime_auth, token_scopes, host_validation, observability_logging — 6+ модулей.

### web.py (1453 строки) — CRITICAL severity

**God-module. Уже запланирован рефакторинг в этапе 59, но масштаб проблем шире:**

1. **Presentation + Business Logic смешаны.** `_render_account_page()` — 290 строк HTML с бизнес-логикой внутри (условия по статусу интеграции, health check).

2. **Паттерн form handler повторяется 7 раз:**
   - register_submit, login_submit, account_integration_submit, account_integration_reauth_submit, account_token_submit, account_token_revoke
   - Каждый: form read → CSRF → rate limit → business call → re-render

3. **Account ID extraction + auth check дублируется 6 раз.**

4. **Dashboard loading содержит бизнес-логику** (822–897): агрегация 7 запросов, трансформация данных, оценка health — должно быть в сервисном слое.

5. **Связность с 15+ модулями** — storage, storage_models, web_auth, request_auth, runtime_auth, service_token_service, vetmanager_connection_service, web_security, observability_logging, service_metrics, request_context, landing_page, exceptions, token_cleanup.

## 61.2 Ревью tools/ — дублирование паттернов

### Массивное дублирование CRUD-boilerplate — CRITICAL

Каждый из 12 tool-модулей независимо реализует идентичные паттерны:

**list:** `VetmanagerClient()` → `build_list_query_params()` → `vc.get(endpoint, params)` — 12 раз.

**by_id:** `VetmanagerClient()` → `vc.get(f"{endpoint}/{id}")` — 12 раз.

**create:** `VetmanagerClient()` → собрать payload dict → `vc.post(endpoint, json=payload)` — 10 раз.

**update:** `VetmanagerClient()` → условный payload (`if field: payload[field] = field`) → `vc.put()` — 12 раз.

**delete:** `VetmanagerClient()` → `vc.delete(f"{endpoint}/{id}")` — 5 раз.

Итого: **50+ функций** с идентичной структурой, отличающихся только endpoint и набором полей.

### Дублирование pagination loop — HIGH

Ручной `while True` + offset + totalCount повторяется в 3 местах:
- `client.py:69–111` (get_debtors)
- `invoice.py:86–112` (get_average_invoice)
- `medical_card.py:102–130` (get_medical_cards_by_client_id)

### Непоследовательное именование полей — MEDIUM

Одни модули используют snake_case, другие camelCase для API-полей:
- `client.py` → `first_name`, `last_name` (snake_case)
- `clinical.py` → `petId`, `doctorId`, `dateIn` (camelCase)
- `finance.py` → `clientId`, `cassaId` (camelCase)
- `operations.py` → `doctor_id`, `clinic_id` (snake_case)

**Причина:** API Vetmanager сам непоследователен — часть endpoints принимает snake_case, часть camelCase. Это не наш баг, а отражение upstream API.

### Непоследовательная обработка optional fields — MEDIUM

Разные sentinel-значения для "не задано":
- `pet.py` → `0` означает "не передавать"
- `good.py:89`, `user.py:90` → `-1` как sentinel
- `client.py` → `None` / пустая строка

### VetmanagerClient() instantiation — MEDIUM

`vc = VetmanagerClient()` создаётся в каждой функции (50+ раз). Client stateless — можно использовать module-level или singleton.

### Дублирование profile/aggregation — MEDIUM

`client.py:227–303` (get_client_profile) и `pet.py:149–225` (get_pet_profile) — идентичный паттерн: fetch main entity + related entities + return dict. Нет общего кода.

## 61.3 Ревью storage layer

### Криптография в моделях — HIGH

`storage_models.py:78–86` — модели VetmanagerConnection и ServiceBearerToken напрямую вызывают `encrypt_secret_payload()`, `decrypt_secret_payload()`, `hash_bearer_token()`. Модели должны быть dumb data structures; crypto — в сервисном слое.

### Непоследовательный доступ к encryption key — HIGH

Два способа получения ключа:
1. `get_storage_encryption_key()` в secret_manager — с валидацией
2. `os.environ.get("STORAGE_ENCRYPTION_KEY")` напрямую в runtime_auth.py и web.py (5 мест) — без валидации

Прямой доступ обходит проверку формата ключа.

### Circular dependency (managed) — LOW

`secret_manager.py:22` — локальный импорт `from web_auth import get_web_session_secret` с комментарием "local import to avoid cycle". Работает, но хрупко.

### Implicit query logic — MEDIUM

`bearer_auth.py:140–145` — `order_by(...).limit(1)` подразумевает "первое соединение = активное". Бизнес-правило закопано в SQL-запросе.

## 61.4 Ревью auth chain

### Валидация домена размазана по 3 модулям — HIGH

- `runtime_auth.py:38–43` — определяет `_validate_domain()` (приватная функция)
- `vetmanager_connection_service.py` — импортирует и вызывает её 9 раз
- `runtime_auth.py:58` — вызывает повторно при каждом запросе (уже валидировано при сохранении)

Приватная функция с `_` используется как публичная. Двойная валидация: при сохранении + при каждом runtime-запросе.

### Scope checking на неправильном уровне — HIGH

Токен валидируется в `bearer_auth.py`, но scope checking происходит позже в `vetmanager_client.py:212–220`. Токен с недостаточными правами проходит auth и отклоняется только при выполнении запроса. Нарушение fail-fast.

### Hidden coupling: auth_audit → web_security — MEDIUM

`auth_audit.py:13` импортирует `resolve_client_ip` из `web_security`. IP-резолв (proxy-aware) живёт в web-слое, но вызывается из auth-слоя.

### Inconsistent error recording — MEDIUM

Auth failures записываются двумя способами:
- `record_auth_failure()` — в метрики (request_auth, bearer_auth, web_security)
- Audit logs — в auth_audit (без метрик)

Нет единого правила, какие ошибки попадают в метрики, а какие только в audit.

### Token lifecycle split — LOW

Методы `is_revoked()`, `is_expired()`, `sync_status()` на модели ServiceBearerToken мутируют состояние. Часть lifecycle в модели, часть в bearer_auth.py.

### Process-local rate limiter — LOW (known)

`bearer_rate_limiter.py` — per-process dict. Не масштабируется на multi-process. Задокументировано, приемлемо для single-instance.
