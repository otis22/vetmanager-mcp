# Этап 172.2. Report AI save narrow scope

## Цель

Сделать `save_report_ai_job_as_report` доступным без Full access и без широкого `analytics.write`, выделив отдельный минимальный scope и preset для сохранения Report AI отчёта.

## Контекст

Production feedback `#10` показал, что для сохранения Report AI отчёта пользователю пришлось выдать токену Full access. Текущий контракт слишком широкий:

- `save_report_ai_job_as_report` в `TOOL_REQUIRED_SCOPES` требует `analytics.write`;
- `POST /rest/api/report-ai-job/{id}/save` в `required_scope_for_request()` требует `analytics.write`;
- `read_only` preset содержит только `analytics.read`, поэтому полный Report AI flow `create -> poll -> save -> data` невозможен без full/write-heavy токена.

Stage 170 уже классифицировал `save_report_ai_job_as_report` как explicit write tool, потому что он создаёт persistent report visible in Vetmanager. Значит переводить его в `analytics.read` или добавлять write-возможность в `read_only` нельзя: preset с label `Read only` должен оставаться без upstream mutations. Нужен более узкий write scope и отдельный минимальный preset.

## Source facts

- `token_scopes.py` хранит supported scopes централизованно и валидирует unknown scopes через `normalize_token_scopes()`.
- `tool_access_registry.py` задаёт preset bundles, tool-level preflight mapping and marketed preset coverage.
- `token_scopes.required_scope_for_request()` делает defense-in-depth mapping для REST path `/rest/api/report-ai-job/{id}/save`.
- `README.md` уже документирует explicit save step and preset-based bearer issuance.
- `artifacts/technical-requirements-vetmanager-mcp-ru.md` описывает current token scope model and active presets.

## Scope

1. Добавить supported scope `report_ai.write`.
   - Scope означает только запись persistent Report AI отчёта через save endpoint.
   - Он не заменяет `analytics.write` для прочих analytics mutations, например `create_timesheet`.
2. Добавить preset `report_ai` / label `Report AI`, который содержит:
   - `analytics.read`;
   - `report_ai.write`.
3. Перевести `save_report_ai_job_as_report` на `report_ai.write`.
4. Перевести `required_scope_for_request("POST", "/rest/api/report-ai-job/{id}/save")` на `report_ai.write`.
5. Оставить `read_only`, `frontdesk`, `finance`, `doctor`, `inventory` без нового write scope.
6. Добавить `save_report_ai_job_as_report` в `MARKETED_PRESET_TOOLS[report_ai]`, чтобы access-denial hints and UI-facing preset coverage совпадали с фактическими scopes.
   - `MARKETED_PRESET_TOOLS[report_ai]` должен перечислять весь Report AI flow: `create_report_ai_job`, `get_report_ai_job`, `confirm_report_ai_job_candidate`, `get_report_ai_job_data`, `start_report_export`, `get_report_export_file`, `get_report_ai_job_export`, `save_report_ai_job_as_report`.
7. Сохранить совместимость ранее выданных full-access токенов:
   - persisted scopes snapshot, exactly equal to the old full-access bundle before `report_ai.write`, is treated as full access and expanded to current `SUPPORTED_TOKEN_SCOPES`;
   - arbitrary partial/custom tokens are not grandfathered.
8. Обновить tests and docs:
   - access registry exact preset snapshots;
   - tool preflight denial message for read/custom token without save scope;
   - REST path scope mapping;
   - token issuance preset scope snapshot;
   - README/technical requirements.

## Migration / Compatibility

- Full-access legacy tokens continue to work because legacy no-scope tokens and `full_access` include all supported scopes.
- Previously issued full-access tokens with non-empty old full-access `scopes_json` also continue to work: the exact old full-access snapshot is treated as full access and expanded by deserialization/runtime helper to current `SUPPORTED_TOKEN_SCOPES`.
- Existing custom tokens with `analytics.write` but without `report_ai.write` will no longer be allowed to save Report AI jobs. This is intentional: `analytics.write` was too broad and should not grandfather Report AI save silently.
- Existing persisted `read_only` and `frontdesk` tokens are not backfilled and should continue to infer as their original presets because their scope bundles are unchanged.

## Out of Scope

- Менять Vetmanager API или upstream permission model.
- Делать custom per-tool scope builder в UI.
- Переклассифицировать `confirm_report_ai_job_candidate`: он остаётся `analytics.read`.
- Менять `analytics.write` semantics для остальных инструментов.
- Автоматически сохранять Report AI jobs внутри read-only tools.

## Acceptance Criteria

1. `SUPPORTED_TOKEN_SCOPES` contains `report_ai.write`.
2. `TOKEN_PRESET_CHOICES` contains `report_ai` with label `Report AI`.
3. `TOKEN_PRESET_SCOPES[report_ai]` is exactly `analytics.read` + `report_ai.write`.
4. `save_report_ai_job_as_report` requires only `report_ai.write` at MCP tool preflight.
5. `POST /rest/api/report-ai-job/{id}/save` requires `report_ai.write` in request-level mapping.
6. `read_only`, `frontdesk`, `finance`, `doctor`, and `inventory` presets do not include `report_ai.write`.
7. `MARKETED_PRESET_TOOLS[report_ai]` includes all Report AI flow tools, including `save_report_ai_job_as_report`.
8. `get_presets_allowing_tool("save_report_ai_job_as_report")` returns `Full access` and `Report AI`.
9. A token with `analytics.read` only is rejected before upstream save call and error mentions `report_ai.write`.
10. A token with `analytics.write` but not `report_ai.write` is rejected before upstream save call.
11. A token with `report_ai.write` can call `save_report_ai_job_as_report` without `analytics.write`.
12. A previously issued full-access token with the exact old full-access scope snapshot is expanded to include `report_ai.write`.
13. Full test suite remains green.
