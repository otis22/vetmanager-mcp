# PRD — Этапы 196, 197, 199: Activation UX (мобильная интеграция, one-click токен, activation-first кабинет)

> **Статус документа: as-built (ретроспективный).** Этапы выполнены 2026-07-10
> одной сессией по явному указанию пользователя «отходи от воркфлоу»: без
> предварительного PRD, без ревью сторонней моделью, без deploy/push.
> Документ фиксирует фактический scope, декомпозицию и проверенные acceptance
> criteria постфактум, чтобы артефакты проекта оставались полными.
> Отклонения задокументированы в `AssumptionLog.md` (запись от 2026-07-10).

## Контекст и проблема

Prod-аналитика 2026-07-10 (30 дней): 5 новых аккаунтов → 4 подключили
Vetmanager → 3 выпустили токен → 1 сделал MCP-запрос. Реальный кейс дня:
9 подряд `POST /account/integration` со статусом 400 с мобильного браузера,
затем успех с десктопа; после выпуска токена — 0 MCP-запросов.

Design/вёрстка review (лог
`LiveHelperAgent/logs/mcp/2026-07-10-activation-funnel-design-review.md`)
выявил механизмы: (1) ошибка интеграции рендерится в середине длинной
страницы и невидима после POST-перезагрузки на мобильном; (2) строгий
lowercase-паттерн домена + автокапитализация мобильной клавиатуры дают 400;
(3) пустая IP-маска молчаливо привязывает токен к IP браузера — первый запрос
клиента с другого IP получает молчаливый 401; (4) инструкция подключения
клиента спрятана в свёрнутый `details`.

## Цель

Новый пользователь проходит путь «регистрация → рабочая интеграция →
выпущенный токен → первый MCP-запрос» без невидимых стен, в том числе
с мобильного.

## Scope и декомпозиция

### Этап 196 — шаг подключения Vetmanager (мобильный)

- 196.2 Нормализация домена server-side в `domain_validation.validate_domain`:
  trim + lower + срез `https://`/`http://`, path/query, суффиксов
  `.vetmanager.ru` и `.vetmanager.cloud`. Мобильные атрибуты инпутов
  (`autocapitalize="none"`, `autocorrect="off"`, `spellcheck="false"`) +
  подсказка формата под полем.
- 196.3 Видимость ошибки: `role="alert"`, `data-autoscroll` + scrollIntoView
  после рендера, русские тексты с конкретным следующим шагом
  (`web_routes_account._integration_error_text` — маппинг на display-слое,
  исключения сервисного слоя остаются английскими для логов/метрик).
- 196.4 Успех ведёт дальше: scroll к плашке + CTA-якорь «Выпустить Bearer
  token» на секцию токенов.
- 196.5 Анти-даблсабмит (`data-submit-lock`, блокировка кнопок, статус
  «Проверяем подключение…») + show/hide toggle для API key.
- 196.6 Reauth-кнопка только при `reauth_required`; подсветка выбранной
  auth-mode карточки (`:has(input:checked)`); disabled-стили.

### Этап 197 — one-click выпуск токена и первый MCP-запрос

- 197.2 Панель «Быстрый старт» в состоянии `needs_token`: дефолты
  (имя «Мой первый токен», 30 дней, Analytics) + явный radio-выбор IP:
  «Работать с любого IP» (guided-дефолт с честным предупреждением) или
  «Только мой текущий IP». Полная форма — в `details` «Настроить вручную».
- 197.3 При `needs_client_use` инструкция подключения — основной блок
  (per-client: Cursor/Claude Code, ChatGPT); на панели выпущенного токена —
  раскрытая инструкция + «Скопировать готовый конфиг» (JSON с URL и токеном);
  endpoint `POST /account/telemetry/token-copied` (сессия + CSRF) →
  business event `token_copied` (добавлен в allowlist).
- 197.4 Индикатор «Ждём первый запрос…» + `GET /account/activation-status`
  (JSON) + polling каждые 15 с с авто-перезагрузкой при смене состояния.

### Этап 199 — activation-first кабинет

- 199.2 Stepper «Шаг N из 3» до состояния `ready`; крупные секции — `<details
  class="section-block">`, открытые только для текущего шага воронки или при
  наличии сообщения (ошибка/успех принудительно раскрывают свою секцию);
  контент остаётся в DOM; якорные ссылки раскрывают целевую секцию (JS
  `openForHash`). Общий helper `compute_activation_state` used страницей и
  polling-endpoint'ом.

## Acceptance criteria (проверены)

1. `validate_domain("Https://Myclinic.vetmanager.ru/") == "myclinic"`;
   невалидные формы по-прежнему отклоняются. ✓ unit + route-тест + браузер.
2. Ошибка интеграции на мобильном видна без скролла (autoscroll + alert),
   текст русский с next-step. ✓ тесты + браузерный прогон 390px.
3. Успех сохранения ведёт CTA в секцию токенов, секция раскрывается якорем. ✓
4. Повторные тапы по submit блокируются на время probe. ✓
5. One-click выпуск с «любого IP» записывает `allowed_ip_mask="*.*.*.*"`
   (warning-лог этапа 155 срабатывает), «текущий IP» — конкретный IP;
   пустое имя → «Мой первый токен». ✓ route-тесты с проверкой БД.
6. `token_copied` доходит до `/metrics` после клика по копированию. ✓ живой
   браузер + `/metrics`.
7. При `needs_client_use` инструкция — главный блок, индикатор ожидания
   активен; после первого запроса страница сама перезагружается в
   «Готово к работе». ✓ живой браузер (симуляция usage в БД, авто-reload).
8. Ни одно состояние воронки не переполняет 390/760/1024px, в том числе со
   всеми раскрытыми секциями и с панелью выпущенного токена. ✓ viewport-тесты.
9. Полная сьюта зелёная: `docker compose --profile test run --rm test` —
   1340 passed, 2 skipped. ✓

## Out of scope

- Продуктовые события failed integration attempts в БД и Grafana-funnel по
  новым аккаунтам — этап 198 (`todo`).
- Изменение дефолтов ручной формы токена (ip_mask prefill текущим IP в
  advanced-форме сохранён).
- Локализация ошибок формы выпуска токена (валидационные ValueError остаются
  английскими).

## Rationale для выбранной сложности

- Нормализация внутри `validate_domain` (а не на каждом call-site) — одна
  точка, все вызовы получают поведение автоматически; строгий client-side
  `pattern` отвергнут, чтобы клиент не отклонял ввод, который сервер теперь
  принимает.
- Guided-дефолт «любой IP» в quick-панели не ослабляет контракт этапа 155:
  ip_mask передаётся явно, выбор виден пользователю вместе с предупреждением,
  wildcard-warning в сервисе сохраняется. Молчаливый IP-lock показал себя как
  activation trap (см. AssumptionLog).
- Сворачивание секций через `<details>`/`open` вместо условного рендера —
  контент остаётся в DOM (тесты, якоря, формы адресуемы), нулевой JS для
  базового поведения.
