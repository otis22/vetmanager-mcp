# PRD: Этап 24. Web-слой, регистрация и кабинет аккаунта

## Цель

Добавить первый внешний пользовательский контур сервиса: лендинг, регистрацию,
login/logout и кабинет account, в котором можно настроить Vetmanager
интеграцию и управлять Bearer-токенами без ручной работы с БД.

## Проблема

- Bearer runtime уже работает, но выпуск account и bearer token пока остаётся
  internal/dev provisioning path.
- Для реального продукта этого недостаточно: пользователь не может сам
  зарегистрироваться, подключить клинику и получить токен для Cursor или
  другого MCP-клиента.
- Следующие этапы с usage analytics и дополнительными auth modes требуют
  устойчивого web-контура, а не только storage и runtime слоёв.

## Границы этапа

- Этап 24 не меняет bearer-only MCP runtime-контракт.
- Поддерживается только один Vetmanager auth mode в кабинете:
  `domain + rest_api_key`.
- Расширенная usage-аналитика Bearer-токенов не входит в этап 24 и остаётся
  этапу 25.
- Второй Vetmanager auth mode (`user login/password -> user token`) не входит в
  этот этап и остаётся этапу 26.

## Пользовательский сценарий первой итерации

1. Пользователь открывает лендинг и понимает модель продукта.
2. Пользователь регистрирует account и входит в кабинет.
3. В кабинете пользователь настраивает активное Vetmanager-подключение через
   `domain + rest_api_key`.
4. Пользователь выпускает Bearer-токен с именем и сроком действия.
5. Пользователь копирует raw token один раз и вставляет его в локальный
   `mcp.json` как `Authorization: Bearer <service_token>`.

## Декомпозиция

### 24.1 PRD и UX-рамка
- Зафиксировать границы web-слоя, роли экранов и порядок пользовательского
  пути.

### 24.2 Лендинг
- Реализовать публичную страницу с описанием сервиса, bearer-only модели и
  supported Vetmanager connection mode.

### 24.3 Регистрация и login/logout
- Добавить обязательную регистрацию account и базовую web-сессию.

### 24.4 Vetmanager integration screen
- Реализовать форму настройки `domain + rest_api_key` с connection test и
  сохранением active connection аккаунта.

### 24.5 Выпуск Bearer-токенов
- Реализовать создание токена с именем и сроком действия.

### 24.6 Список токенов
- Показать список токенов аккаунта со статусом, сроком действия, последним
  использованием и количеством запросов.

### 24.7 Одноразовый показ raw token
- Зафиксировать и реализовать правило one-time reveal: raw bearer показывается
  только сразу после создания и больше не хранится в читаемом виде.

## Требования к UX и безопасности

- Лендинг и кабинет должны говорить на языке bearer-only продукта, без legacy
  `X-VM-*` терминологии.
- Кабинет должен объяснять пользователю, что `domain` и Vetmanager `api_key`
  остаются внутри сервиса и не передаются в MCP client config.
- Web-слой не должен логировать секреты и не должен повторно раскрывать raw
  bearer после завершения шага создания.
- Список токенов должен использовать только безопасные поля (`name`, `status`,
  `expires_at`, `last_used_at`, `request_count`, `token_prefix`).

## Критерии готовности для 24.1

- В проекте есть отдельный PRD этапа 24.
- PRD фиксирует границы первой web-итерации и не смешивает её с этапами 25 и 26.
- Из PRD понятно, какие экраны и пользовательские действия должны появиться в
  задачах `24.2`–`24.7`.
