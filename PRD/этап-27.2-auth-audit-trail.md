# PRD: Этап 27.2. Более подробный audit trail по auth events

## Цель

Сделать аудит Bearer-аутентификации и связанных security-sensitive действий
более подробным, чтобы сервис фиксировал не только lifecycle токена
(`token_created`, `token_revoked`), но и фактические auth events в runtime.

## Контекст

- Сейчас `TokenUsageLog` уже существует и хранит:
  - `bearer_token_id`
  - `event_type`
  - `event_at`
  - `ip_address`
  - `user_agent`
  - `details_json`
- Но фактически логируются только:
  - `token_created`
  - `token_revoked`
- Bearer runtime (`resolve_bearer_auth_context`) пока не пишет audit events ни
  для успешной аутентификации, ни для безопасных отказов по найденному токену.

## Граница текущего решения

На этом шаге не добавляется новая БД-схема и не вводится отдельная generic
таблица для всех возможных auth failures.

Решение ограничивается token-centric audit trail:
- логируются события, которые можно надёжно привязать к существующему
  `bearer_token_id`;
- неизвестный Bearer-токен (`invalid bearer token` до lookup token record)
  отдельно не логируется, потому что в текущей схеме нет сущности, к которой его
  можно безопасно и корректно привязать.

## События, которые должны появиться

- `token_auth_succeeded`
- `token_auth_failed_revoked`
- `token_auth_failed_expired`
- `token_auth_failed_no_connection`
- `token_auth_rate_limited`

Существующие события сохраняются:
- `token_created`
- `token_revoked`

## Данные, которые должны попадать в лог

- `event_type`
- `event_at`
- `bearer_token_id`
- `ip_address` и `user_agent`, если текущий request context их содержит
- безопасные `details_json`, например:
  - `account_id`
  - `connection_id`
  - `auth_mode`
  - `domain`
  - `reason`
  - `retry_after_seconds`

## Что не должно попадать в лог

- raw Bearer token
- `token_hash`
- Vetmanager secret (`api_key`, `user_token`)
- пароль account-а

## Декомпозиция

### 27.2.1 Общий audit helper
- Вынести общий helper записи `TokenUsageLog`.
- Централизовать сериализацию details и извлечение request metadata.

### 27.2.2 Runtime auth events
- Логировать успешную Bearer-аутентификацию.
- Логировать отказные ветки, где токен найден:
  - revoked
  - expired
  - no connection
  - rate limited

### 27.2.3 Lifecycle metadata enrichment
- Перевести `token_created` и `token_revoked` на общий helper, чтобы они тоже
  получали `ip_address` / `user_agent` при вызове из web-контура.

### 27.2.4 Tests
- Добавить unit tests на runtime auth events.
- Добавить web tests на metadata для create/revoke logs.

## Критерии готовности

- `TokenUsageLog` получает auth success/failure events по Bearer runtime.
- `TokenUsageLog` получает request metadata там, где есть HTTP request context.
- Логи не содержат raw токены или другие секреты.
- Тесты покрывают success/failure auth events и metadata enrichment.
