# PRD: Этап 42.3. Deterministic upstream mocks для browser auth flows

## Контекст

Этап 42.2 уже дал browser tests живой localhost HTTP harness. Следующий блокер
для настоящих browser happy-path сценариев — внешние запросы к Vetmanager:
- billing host resolve;
- API-key validation;
- login/password -> user token exchange;
- проверка валидности user token.

Если эти запросы останутся реальными, default browser suite будет нестабильным
и зависимым от сети/внешнего тестового контура.

## Цель 42.3

Подготовить детерминированный mock layer для обоих поддерживаемых auth flow,
который будет использоваться browser tests поверх live localhost harness.

## Входит

- Общие pytest fixtures/helpers для upstream mocks.
- Покрытие обоих auth modes:
  - `domain + api_key`
  - `login/password -> user token`
- Детерминированные ответы на billing resolve и целевые Vetmanager endpoints.
- Тесты/проверки, что mock fixtures дают ожидаемый контракт для browser tests.

## Не входит

- Полный browser happy-path по UI.
- Cleanup test accounts.
- Optional real browser mode.

## Решения

- Использовать `respx` как process-global patcher `httpx`, чтобы live HTTP
  server и его внутренние исходящие запросы также попадали в mock layer.
- Для `domain_api_key` flow мокать:
  - `GET https://billing-api.vetmanager.cloud/host/{domain}`
  - `GET {resolved_host}/rest/api/user`
- Для `user_token` flow мокать:
  - `GET https://billing-api.vetmanager.cloud/host/{domain}`
  - `POST {resolved_host}/token_auth.php`
  - `GET {resolved_host}/rest/api/user`
- Ответы должны быть стабильными и минимально достаточными для login/integration
  happy-path browser tests.

- Browser/live tests могут включить deterministic upstream mocks для обоих auth modes.
- Mocks не требуют внешнего Vetmanager.
- Full suite проходит с новым mock layer.
