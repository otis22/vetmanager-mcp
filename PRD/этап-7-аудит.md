# PRD Этап 7: Аудит полноты реализации

## Цель
Сверить все реализованные MCP-инструменты со справочником сущностей API.
Выявить пробелы в покрытии. Зафиксировать матрицу в AssumptionLog.

## Задачи

### 7.1–7.2 Сверка сущностей и операций
Проверить все 35 сущностей из `api_entity_reference-ru.md`.
Для каждой — какие операции (GET list, GET by id, POST, PUT, DELETE) реализованы.

### 7.3 Недостающие операции
Добавить PUT (update) для сущностей, где это имеет практический смысл:
- `Client` — уже есть `update_client`
- `Pet` — добавить `update_pet`
- `Admission` — добавить `update_admission`
- `MedicalCard` — добавить `update_medical_card`

### 7.4 Тестовое покрытие
Проверить, что существующие тесты охватывают ключевые пути.

### 7.5 Матрица покрытия
Записать итоговую матрицу в AssumptionLog.md.

## Критерии готовности
- Матрица сущность × операция зафиксирована
- Все пробелы задокументированы или закрыты
- AssumptionLog обновлён

## Пометка по Variant A (плановая)
- В последующих этапах аудита учитывать целевую схему credentials:
  `mcp.json` клиента (`url` + `headers`) как источник runtime `domain`/`api_key`.
- Проверять, что тестовые credentials (`TEST_DOMAIN`/`TEST_API_KEY`) не смешиваются
  с рабочим контуром и используются только в e2e real tests.
