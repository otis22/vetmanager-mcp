# Правила работы Claude Code над проектом

Этот файл — основной источник инструкций для Claude Code в данном проекте.
Workflow адаптирован из `.cursor/rules/agent-workflow.mdc` с добавлением обязательного Codex-ревью.

---

## 1. Workplan

- Workplan хранится в `Roadmap.md` — единственный источник очереди работ
- Формируется на основе PRD
- Содержит только название этапа/задачи и статус (`todo` / `in_progress` / `done` / `stop`)
- Не содержит деталей реализации

---

## 2. Выбор задачи

- Только задачи со статусом `todo`
- Без незавершённых зависимостей
- Самая верхняя в списке

Запрещено:
- брать несколько задач одновременно
- перескакивать через задачи
- создавать задачи вне Workplan

---

## 3. PRD задачи

Перед началом реализации:

- Создать PRD в `/PRD/`
- Выполнить декомпозицию: подзадача ≤ 2 ч или ≤ 150 строк кода
- Без декомпозиции реализация запрещена

---

## 4. Core Loop

Агент работает строго по циклу:

1. **Выбрать задачу** из Roadmap
2. **Короткий ресёрч** — изучить контекст, артефакты, API-спецификацию
3. **Написать тесты** (test-first)
4. **Red → Green** — реализовать до прохождения тестов
5. **Запустить все проверки** — `docker compose --profile test run --rm test`
6. **Аудит изменений:**
   - legacy-паттерны, дублирование, рассинхрон с контрактом
   - при необходимости — рефакторинг
7. **Повторный прогон** тестов после аудита/рефакторинга (если были изменения)
8. **Codex-ревью** (см. раздел ниже) — на staged/uncommitted изменениях, **перед commit**, с PRD задачи в контексте
9. **Оценка адекватности замечаний** — для каждого finding решить, относится ли оно к scope текущей задачи и реальной проблеме (см. раздел 5.2)
10. **Устранение адекватных замечаний** — исправить → тесты → повторное codex-ревью (цикл до 0 адекватных critical)
11. **Commit** — только после чистого ревью
12. **Push**

### Запрещено:
- писать код без тестов
- коммитить без прохождения Codex-ревью
- пушить без прохождения проверок
- пропускать повторный прогон после рефакторинга

---

## 5. Codex-ревью перед Commit

Обязательный шаг **перед commit**. Выполняется через скилл `/codex rescue` на незакоммиченных изменениях.

### 5.1 Порядок вызова

**Важно — ограничение sandbox**: агент `/codex rescue` запускается в network-isolated sandbox и **часто не может читать файлы из репозитория** (ошибка `bwrap: loopback: Failed RTM_NEWADDR: Operation not permitted`). Поэтому промпт должен быть **самодостаточным** — весь контекст для ревью передаётся inline в теле промпта, без расчёта на то, что Codex откроет файлы.

Если первый запуск вернул сообщение вида «couldn't read files» / «sandbox error» / пустой результат — это всегда **sandbox fail, не технический refusal**. Перезапустить с diff'ом inline.

1. После прохождения тестов и аудита, **перед commit**, собрать полный контекст:
   ```bash
   git diff                    # все не-закоммиченные изменения
   git diff --cached            # staged
   git status --short           # список новых файлов
   ```
   Для **новых файлов** (появляются в `git status --short` как `??`) — включить в промпт полностью или ключевые фрагменты, потому что `git diff` их не покажет.

2. Вызвать `/codex rescue` с промптом, который **обязательно** содержит:
   - **PRD задачи** (полный текст из `/PRD/{stage}.md` или 5–10 строк описания цели, scope, ограничений)
   - **Ключевые факты про API-контракт** (имена полей, эндпоинты, enum'ы), проверенные на этапе ресёрча — Codex не сможет сам открыть `artifacts/vetmanager_openapi_v6.json`
   - **Полный diff** (`git diff` output) inline в fenced code block
   - **Содержимое новых файлов** (или ключевых их частей) inline, потому что в `git diff` их нет
   - **Test results** с количеством passed/failed
   - **Явные вопросы** для ревью — что проверять
   - **Явное указание** «не трогай файловую систему, весь контекст уже в промпте» — это снимает попытки Codex читать локальные файлы и падать на sandbox

   Шаблон промпта:
   ```
   Code review uncommitted changes. Task: {краткое название}.
   Do NOT touch the filesystem — all context is inline in this prompt.

   PRD (scope and intent):
   {содержимое PRD/{stage}.md или 5-10 строк описания цели,
    подзадач, ограничений}

   API contract facts (verified during research phase, not re-verifiable in this session):
   - {endpoint 1}: filterable fields — {field names, types, operators}
   - {endpoint 2}: enum {name}: {values}
   - {важные связи, mapping внешних имён во внутренние}

   === DIFF ===
   ```diff
   {полный вывод `git diff` здесь}
   ```

   === NEW FILE: {path} ===
   {содержимое нового файла целиком или ключевая часть}

   Test results: {N passed, M failed, suite command}

   === REQUEST ===
   Review for:
   1. {конкретный аспект 1}
   2. {конкретный аспект 2}
   3. {edge cases, о которых я переживаю}

   Format findings as: severity (critical/warning/nit), file:function, description.
   Keep under {300-500} words.
   ```

3. Если всё равно sandbox-ошибка — считать первую попытку не состоявшейся (не тратит лимит итераций), перезапустить с более компактным inline-контекстом. Если sandbox стабильно падает третий раз подряд — сделать audit руками и задокументировать в AssumptionLog пропуск codex-ревью с обоснованием.

### 5.2 Оценка адекватности замечаний

Codex может предлагать улучшения, которые **выходят за scope текущей задачи** или относятся к гипотетическим проблемам. Перед исправлением каждого замечания провести оценку:

| Критерий | Адекватно | Неадекватно |
|----------|-----------|-------------|
| **Scope** | Относится к коду в diff | Касается legacy кода или соседних модулей |
| **Реальность** | Указывает на конкретную ошибку или риск | Гипотетический "может быть в будущем" |
| **PRD** | Соответствует целям этапа | Расширяет задачу за пределы PRD |
| **ROI** | Fix маленький, риск регрессии низкий | Большой рефакторинг для nit |

**Адекватные** — исправлять обязательно (если critical) или по приоритету (warning).
**Неадекватные** — задокументировать в commit message или AssumptionLog с обоснованием отказа, не делать.

### 5.3 Severity reaction

- **critical** (адекватный) — обязательно исправить → тесты → повторное codex-ревью (см. лимит в 5.4)
- **critical** (неадекватный) — задокументировать почему отклонено в AssumptionLog, продолжить
- **warning** (адекватный) — исправить если разумно
- **warning** (неадекватный) — пропустить с пояснением
- **nit** — на усмотрение, можно пропустить

### 5.4 Лимит итераций ревью

Максимум **2 запуска codex-ревью на одну задачу** (1 первичный + 1 повторный).

Если после 2 итераций всё ещё остаются адекватные critical:
- **STOP**: не запускать ревью в 3-й раз
- Задокументировать оставшиеся findings в AssumptionLog с пометкой "deferred after 2 codex iterations"
- Решить: либо вернуться к задаче в отдельном этапе, либо закоммитить с явным acknowledgement технического долга
- Спросить пользователя если непонятно как поступить

Цель лимита: избежать infinite loop, когда каждая итерация Codex генерирует новые findings вместо подтверждения чистоты кода.

### 5.5 Когда можно пропустить codex-ревью

- Изменения только в документации (`.md` файлы, без кода)
- Изменения только в конфигах CI/CD без логики

## 5a. Deep Review — `/super-review` (по требованию)

Многоплановый ревью через skill `/super-review`. **Запускается только по явному запросу пользователя** — не в автоматическом цикле, не после каждого merged этапа.

**Когда запускать (только по явному «сделай ревью» / «проверь»):**
- Пользователь просит проверить накопленный за N этапов drift
- Перед релизом (если пользователь этого хочет)
- Для baseline audit при входе в новый крупный кусок работы
- Когда есть подозрение на systemic проблему, которая могла ускользнуть от per-commit Codex

**Агент НЕ должен:**
- Запускать `/super-review` сам после merge этапов
- Включать deep-review в Core Loop per-stage workflow (раздел 4) — там остаётся только per-commit Codex (раздел 5)
- Предлагать deep-review «на всякий случай» — это дорого (9 параллельных агентов + Codex), запускается осознанно

**Что запускается (в параллель):**
- 8 специализированных ревьюеров: `code`, `architecture`, `docs`, `security`, `performance-and-reliability`, `observability`, `tests`, `product` + 9-й `codex-blindspot` (параллельный Codex pass) + `aggregator` (10-й subagent, оценивает адекватность и формирует итоговый отчёт) = 10 subagent'ов в `.claude/agents/`
- 9-й — `codex-blindspot`: один Codex-пробег с анти-корреляционным промптом (ищет то, что Claude-модели системно пропускают)
- `scripts/review_workflow_check.sh` — механические проверки (PRD файл, Roadmap статус, AssumptionLog, diff size)
- Агрегатор (`reviewer-aggregator`) — дедупликация, ранжирование, итоговый verdict
- Финальный Codex-арбитраж на top-10 blocker+high findings

**Scope (аргумент skill):**
- `changed` (дефолт) — `git diff` с main + uncommitted
- `related` — changed + 1 уровень импортов
- `full` — весь кодбейз
- `stage:N` — файлы из PRD/этап-N-*.md

**Codex-интеграция:**
- Default on (флаг `--no-codex` для отключения)
- Каждый субагент может эскалировать **до 2 findings** с confidence 0.4-0.7 в Codex для валидации
- Финальный арбитраж — один промпт с inline snippets top-10 (≤ 50k токенов)
- Fallback при sandbox fail: retry × 1, затем секция «codex skipped» в отчёте

**Отчёт:** `artifacts/review/{YYYY-MM-DD}-{scope}-stage-{N}.md`, коммитится с остальным кодом.

**Файлы инфраструктуры:**
- `.claude/agents/reviewer-*.md` — определения 10 агентов
- `.claude/commands/super-review.md` — skill
- `scripts/review_workflow_check.sh` — mechanical checks

  - неясности

Задача считается завершённой только после записи в AssumptionLog.

## 9. Тесты и проверки

- Unit + mock e2e: `docker compose --profile test run --rm test`
- Real API (нужны `TEST_DOMAIN`, `TEST_API_KEY`): `docker compose --profile test run --rm -e TEST_DOMAIN=<домен> -e TEST_API_KEY=<ключ> test`
- CI: `.github/workflows/test.yml` (unit + mock); `test-real.yml` — ручной запуск

### 9.1 Изменения зависимостей → обязательный rebuild

Если в commit меняется любой из файлов:
- `Dockerfile` (особенно секции `pip install`, добавление/удаление upper bounds)
- `requirements.txt` / `pyproject.toml` (если появятся)
- `docker-compose.yml` (image config)

то **обязательно** перед прогоном тестов сделать:
```bash
docker compose --profile test build test
```

**Почему**: docker layer cache может содержать старую версию зависимости. Без rebuild тесты могут "проходить" на устаревшем образе, скрывая регрессию (например, upper bound может откатить production-версию пакета на старую с breaking changes).

**Кейс**: этап 58 добавил `fastmcp<3`, что молча откатило 3.1.1 → 2.14.6 при следующем rebuild. 26 тестов сломались, потому что fastmcp 2.14.x убрал public `call_tool` с FastMCP instance. Регрессия была скрыта 4 коммита, пока test image не пересобрали.

### 9.2 Pre-commit чеклист для dependency changes

Перед commit любого изменения зависимостей:
1. Узнать **текущую** установленную версию: `docker run --rm <prod_image> pip show <package>`
2. Убедиться что новый pin **включает** текущую версию (не откатывает)
3. `docker compose --profile test build test` — fresh rebuild
4. `docker compose --profile test run --rm test` — полный прогон
5. Только после этого — Codex review + commit
