# Правила работы Claude Code над проектом

Этот файл — основной источник инструкций для Claude Code в данном проекте.
Workflow адаптирован из `.cursor/rules/agent-workflow.mdc` с добавлением обязательного Codex-ревью.

---

## 1. Workplan

- Workplan хранится в `Roadmap.md` — единственный источник очереди работ
- Формируется на основе PRD
- Содержит только название этапа/задачи и статус (`todo` / `in_progress` / `done` / `stop`)
- Не содержит деталей реализации

---

## 2. Выбор задачи

- Только задачи со статусом `todo`
- Без незавершённых зависимостей
- Самая верхняя в списке

Запрещено:
- брать несколько задач одновременно
- перескакивать через задачи
- создавать задачи вне Workplan

---

## 3. PRD задачи

Перед началом реализации:

- Создать PRD в `/PRD/`
- Выполнить декомпозицию: подзадача ≤ 2 ч или ≤ 150 строк кода
- Без декомпозиции реализация запрещена

---

## 4. Core Loop

Агент работает строго по циклу:

1. **Выбрать задачу** из Roadmap
2. **Короткий ресёрч** — изучить контекст, артефакты, API-спецификацию
3. **Написать тесты** (test-first)
4. **Red → Green** — реализовать до прохождения тестов
5. **Запустить все проверки** — `docker compose --profile test run --rm test`
6. **Аудит изменений:**
   - legacy-паттерны, дублирование, рассинхрон с контрактом
   - при необходимости — рефакторинг
7. **Повторный прогон** тестов после аудита/рефакторинга (если были изменения)
8. **Commit**
9. **Codex-ревью** (см. раздел ниже)
10. **Устранение замечаний** — если есть, исправить → тесты → commit
11. **Push** — только после чистого ревью

### Запрещено:
- писать код без тестов
- пушить без прохождения проверок
- пропускать повторный прогон после рефакторинга
- пушить без прохождения Codex-ревью

---

## 5. Codex-ревью перед Push

Обязательный шаг между Commit и Push. Выполняется через скилл `/codex rescue`.

### Порядок:

1. После commit вызвать `/codex rescue` с промптом:
   ```
   Code review последнего коммита. Проверь:
   - корректность логики и обработки ошибок
   - соответствие API-контракту (artifacts/vetmanager_openapi_v6.json)
   - отсутствие секретов, хардкода, SQL-инъекций
   - качество тестов: покрытие edge cases
   - соответствие стилю и паттернам проекта
   Выдай список замечаний с severity (critical / warning / nit).
   ```
2. Проанализировать ответ Codex:
   - **critical** — обязательно исправить, заново тесты, новый commit, повторное ревью
   - **warning** — исправить если разумно, пояснить если нет
   - **nit** — на усмотрение, можно пропустить
3. Push разрешён только когда нет открытых critical-замечаний

### Когда можно пропустить:

- Изменения только в документации (`.md` файлы, без кода)
- Изменения только в конфигах CI/CD без логики

---

## 6. Завершение задачи

После выполнения:

- Обновить статус в `Roadmap.md`
- Записать в `AssumptionLog.md`:
  - допущения
  - неясности
  - архитектурные решения
  - результат Codex-ревью (если были значимые замечания)

Задача считается завершённой только после записи в AssumptionLog.

---

## 7. Артефакты проекта

Все управленческие артефакты — в Markdown.

| Файл / папка | Назначение |
|--------------|------------|
| `Roadmap.md` | Workplan — этапы, задачи, статусы |
| `/PRD/*.md` | PRD задач с декомпозицией |
| `AssumptionLog.md` | Журнал допущений и решений |

---

## 8. Справочные артефакты (artifacts/)

Обращаться при планировании, декомпозиции и реализации:

| Файл | Когда использовать |
|------|--------------------|
| `artifacts/prd-vetmanager-mcp-ru.md` | Главный PRD: видение, цели, персоны, требования |
| `artifacts/technical-requirements-vetmanager-mcp-ru.md` | Технические требования: стек, архитектура, структура |
| `artifacts/api_entity_reference-ru.md` | Справочник сущностей API |
| `artifacts/vetmanager_openapi_v6.json` | OpenAPI спецификация — источник истины |
| `artifacts/vetmanager_postman_collection.json` | Postman-коллекция — вспомогательный материал |

> Агент **не домысливает** поведение API — всё проверяется по OpenAPI или `api_entity_reference-ru.md`.

---

## 9. Тесты и проверки

- Unit + mock e2e: `docker compose --profile test run --rm test`
- Real API (нужны `TEST_DOMAIN`, `TEST_API_KEY`): `docker compose --profile test run --rm -e TEST_DOMAIN=<домен> -e TEST_API_KEY=<ключ> test`
- CI: `.github/workflows/test.yml` (unit + mock); `test-real.yml` — ручной запуск
