# Правила работы Claude Code над проектом

Этот файл — основной источник инструкций для Claude Code в данном проекте.
Workflow адаптирован из `.cursor/rules/agent-workflow.mdc` с добавлением обязательного ревью сторонней моделью.

---

## 1. Workplan

- Workplan хранится в `Roadmap.md` — единственный источник очереди работ
- Формируется на основе PRD
- Содержит только название этапа/задачи и статус (`todo` / `in_progress` / `done` / `stop`)
- Не содержит деталей реализации

---

## 2. Выбор задачи

- Только задачи со статусом `todo`
- Без незавершённых зависимостей
- Самая верхняя в списке

Запрещено:
- брать несколько задач одновременно
- перескакивать через задачи
- создавать задачи вне Workplan

---

## 3. PRD задачи

Перед началом реализации:

- Создать PRD в `/PRD/`
- Выполнить декомпозицию: подзадача ≤ 2 ч или ≤ 150 строк кода
- Изучить связанные `/artifacts` после создания PRD и обновить PRD проверенными фактами
- Провести PRD-review и PRD-review сторонней моделью
- Провести оценку PRD на простоту и повторить PRD-review gates после правок PRD
- Без декомпозиции реализация запрещена

---

## 3.1. Ревью сторонней моделью и бюджет

`Ревью сторонней моделью` означает:

- если текущий агент работает в Claude, ревью делает Codex `gpt-5.5`;
- если текущий агент работает в Codex, ревью делает Claude Opus.

Бюджеты:

- PRD-review сторонней моделью: максимум 2 запуска на задачу;
- code/diff review сторонней моделью: максимум 2 запуска на задачу;
- бюджеты раздельные и не расходуют друг друга;
- `gpt-5.3-codex-spark` можно запускать как scout/subagent безлимитно; Spark не принимает финальных решений и не расходует бюджет.

В пользовательском тексте и чеклистах используй формулировку **«ревью сторонней моделью»**, а не имя конкретного CLI, кроме Evidence/Notes.

---

## 4. Core Loop

Агент работает строго по циклу:

1. **Выбрать задачу** из Roadmap
2. **Создать / обновить PRD** задачи (см. раздел 3) — scope, подзадачи, acceptance, out-of-scope
3. **Изучить связанные `/artifacts`** — PRD, technical requirements, API notes/reference/OpenAPI и другие файлы по теме задачи
4. **Обновить PRD после artifacts** — внести проверенные API/architecture/product facts и убрать неподтверждённые допущения
5. **PRD-review** — устранить адекватные findings; если PRD изменился, повторить PRD-review до отсутствия адекватных critical/high findings
6. **PRD-review сторонней моделью** — устранить адекватные findings; если PRD существенно изменился, повторить в пределах бюджета PRD-review
7. **Оценка PRD на простоту** (см. раздел 4.1) — если найден проще вариант, переписать PRD; после любых правок повторить PRD-review, после существенных правок повторить ревью сторонней моделью в пределах бюджета
8. **Зафиксировать финальную PRD-версию** — PRD согласован с artifacts, прошёл review gates и содержит rationale для выбранной сложности, если она нетривиальна
9. **Написать тесты** (test-first)
10. **Red → Green** — реализовать до прохождения тестов
11. **Запустить все проверки** — `docker compose --profile test run --rm test`
12. **Аудит изменений:**
   - legacy-паттерны, дублирование, рассинхрон с контрактом
   - при необходимости — рефакторинг
13. **Повторный прогон** тестов после аудита/рефакторинга (если были изменения)
14. **Commit** — только после зелёных проверок и локального аудита
15. **Ревью сторонней моделью на committed diff до push** (см. раздел 5) — устранить адекватные findings; если были изменения, повторить tests/audit/commit/review в пределах бюджета code/diff review
16. **Push** — только после прохождения code/diff review gate или явного исчерпания бюджета с rationale
17. **Workflow self-attestation checklist** (см. раздел 4.2) — заполнить чеклист по всем предыдущим шагам и показать пользователю после push
18. **Завершение задачи** — обновить `Roadmap.md` и `AssumptionLog.md`

### Запрещено:
- писать код без тестов
- коммитить без прохождения тестов и аудита
- пушить без прохождения проверок
- пушить без прохождения ревью сторонней моделью на committed diff либо без явного исчерпания бюджета с rationale
- пропускать повторный прогон после рефакторинга
- пропускать PRD-review gates после правок PRD
- пропускать оценку простоты PRD для задач с нетривиальным кодом
- завершать задачу без заполненного self-attestation чеклиста

### 4.1 Оценка простоты PRD

**Обязательный шаг после artifacts и PRD-review, перед написанием кода.** Цель — поймать over-engineering до реализации, когда изменить направление дёшево, а не после 500 LOC кода.

Задать PRD три вопроса:

1. **Читаемость** — за сколько секунд новый читатель поймёт намерение? Если решение требует длинного prose-объяснения «почему так сложно» в PRD — оно, скорее всего, слишком сложное.
2. **Поддерживаемость** — сколько файлов/мест придётся трогать при разумном изменении требований (новая сущность, новое поле, новый auth mode)? Если > 5 — red flag.
3. **Очевидность** — делает ли реализация ровно то, что обещает её сигнатура, без скрытых side-effects, магических хуков, циркулярных импортов, multi-hop indirection?

**Конкретные triggers на упрощение** (если один из них есть — попробовать simpler вариант):

- **Abstraction без 2+ call-sites**: класс / generic / protocol для одного клиента → замени на функцию / прямой вызов
- **Premature flexibility**: configurable-everything, factory-for-one-type → hardcoded + rename позже если нужно
- **Indirection > 2 hops**: caller → shim → canonical → impl — обычно хватает 1 hop'а или прямого пути
- **Dual-API surface**: "new code should use X, but Y keeps working" — признак незавершённой миграции; либо добей миграцию сейчас, либо оставь только Y
- **Sync mechanisms paired**: manual cache sync + event listener + fallback rebuild — один из трёх обычно лишний
- **State machine > 3 states** там где `is_ready: bool` хватает
- **Lazy imports для не-циклического случая** — чаще всего признак, что разбиение на модули неверное
- **Heavy framework где stdlib достаточно**: pydantic model для 2-поля dict, asyncio.Queue где dict работает
- **Helper вызывается из 1 места** — inline его

**Процедура:**

1. Записать PRD как есть (текущий вариант).
2. Пройтись по triggers выше. Для каждого совпадения — спросить: «а если заменить на ___?».
3. Если simpler вариант реально покрывает все acceptance criteria — переписать PRD. Если не покрывает — зафиксировать в PRD блоком **«Rationale для выбранной сложности»**, почему simpler вариант отклонён (concrete use case, measured performance, security requirement).
4. После любых правок PRD повторить обычный PRD-review.
5. После существенных правок PRD повторить ревью сторонней моделью, если остался бюджет PRD-review.
6. Зафиксировать финальную PRD-версию до того, как начнётся код.

**Критерий пропуска:** шаг можно пропустить только для тривиальных bug fix'ов (< 50 LOC изменений, 0 новых абстракций). Для архитектурных этапов, рефакторингов, новых модулей — обязателен.

**Rationale:** large refactor committed this session (103a/c/d) показал, что без этого шага сложно поймать over-engineering на раннем этапе — BC shim layers, dual API surfaces, multi-hop indirection появляются "естественно" при миграции и остаются в коде как tech debt.

### 4.2 Self-attestation workflow checklist

**Обязательный шаг после push.** Цель — явная самопроверка, что каждый шаг Core Loop действительно выполнен, а не пропущен под шум. Агент показывает заполненный чеклист пользователю после `git push`.

**Формат (Markdown) — показать пользователю дословно, заполнить реальными значениями):**

```markdown
## Workflow self-attestation — этап {N.N} {краткое название}

| # | Step | Status | Evidence / Notes |
|---|------|--------|------------------|
| 1 | Задача выбрана из Roadmap | ✓/✗ | {строка Roadmap, до изменений status} |
| 2 | PRD создан / обновлён | ✓/✗ | {путь к файлу `PRD/этап-N-*.md`} |
| 3 | Связанные artifacts изучены | ✓/✗ | {api-research-notes §X, technical requirements §Y, OpenAPI/API reference, другие источники} |
| 4 | PRD-review выполнен | ✓/✗ | {N findings; N адекватных; повтор после правок если был} |
| 5 | PRD-review сторонней моделью выполнен | ✓/✗/N-A | {модель, запуск 1/2 или 2/2, findings, rationale если budget exhausted/skipped} |
| 6 | Оценка PRD на простоту | ✓/✗/N-A | {triggers проверены; результат; повторные PRD-review gates после правок} |
| 7 | Тесты написаны (test-first) | ✓/✗ | {файлы tests/test_*.py, количество новых тестов} |
| 8 | Red → Green реализация | ✓/✗ | {коротко что реализовано} |
| 9 | Все проверки зелёные | ✓/✗ | {N passed / M failed, команда: `docker compose --profile test run --rm test`} |
| 10 | Аудит изменений выполнен | ✓/✗ | {legacy patterns, duplication, field-name contract} |
| 11 | Повторный прогон после аудита | ✓/✗/N-A | {N passed; или N-A если аудит не потребовал изменений} |
| 12 | Commit выполнен | ✓/✗ | {commit SHA} |
| 13 | Ревью сторонней моделью на committed diff выполнено | ✓/✗/N-A | {модель, запуск 1/2 или 2/2, findings, rationale если budget exhausted/skipped} |
| 14 | Адекватные findings устранены | ✓/✗/N-A | {commits/изменения; либо N-A если 0 адекватных} |
| 15 | Push выполнен | ✓/✗ | {remote/branch, commit SHA} |
| 16 | Этот чеклист заполнен | ✓ | показан пользователю после push |

### Summary
- **Файлов изменено**: X
- **LOC diff**: +Y / -Z
- **Тесты**: N passed
- **Ревью сторонней моделью**: PRD launches X/2, code launches Y/2; blocker=0, high=0 after fixes
- **PRD**: `PRD/этап-N-*.md`
- **Commit message draft**: {одна-две строки}

### Отклонения от workflow
{Если что-то пропущено или отклонение — явно перечислить с обоснованием. Если отклонений нет — «Отклонений нет.»}

Workflow завершён?
```

**Правила заполнения:**

1. **Не пиши ✓, если шаг реально не сделан.** Self-attestation — контракт с пользователем; пустая галочка разрушает доверие к воркфлоу.
2. **Для шагов с N-A** (пропущенных легитимно) — обязательно укажи причину пропуска в Notes (например, «N-A: trivial fix, <50 LOC, 0 новых абстракций» для шага 4).
3. **Evidence должно быть конкретным**: ссылка на файл, строку Roadmap, id/команду ревью сторонней моделью, номер коммита. Не «ок», не «сделано».
4. **Отклонения от workflow** — отдельная секция. Если ты сознательно пропустил ревью сторонней моделью (§5.5 docs-only) или simplicity eval (trivial fix) — это здесь. Пустое поле «Отклонений нет» тоже валидно.
5. Чеклист — **первое, что пользователь видит после `git push`**. Вставляй его в свой текстовый ответ как Markdown-блок, не в коммит-сообщение и не в отдельный файл.

**Критерий пропуска:** чеклист обязателен для всех задач, которые делают push. Исключение: commits, которые меняют ТОЛЬКО текстовую документацию (*.md files, CLAUDE.md), могут использовать сокращённый чеклист — PRD/artifacts/review/test/code шаги отмечаются как N-A со ссылкой на §5.5 docs-only.

**Rationale:** super-review 2026-04-18 показал, что некоторые этапы шли в prod без явного прохождения каждого шага (например, AssumptionLog секция отсутствовала для 103a/c/d, technical-requirements не обновлён). Self-attestation делает такие пропуски видимыми до коммита — пользователь замечает пустую ячейку и говорит «вернись и доделай», вместо того чтобы ловить это через super-review уже в prod.

## 5. Ревью сторонней моделью

Обязательный code/diff gate **после commit и перед push**. Выполняется сторонней моделью на committed diff.

- Если агент работает в Claude, ревью делает Codex `gpt-5.5`.
- Если агент работает в Codex, ревью делает Claude Opus.
- Бюджет: максимум 2 запуска на code/diff review.
- `gpt-5.3-codex-spark` можно запускать как scout/subagent безлимитно; он не расходует бюджет и не принимает финальных решений.

### 5.1 Порядок вызова

**Важно — ограничение sandbox**: агент `/codex rescue` запускается в network-isolated sandbox и **часто не может читать файлы из репозитория** (ошибка `bwrap: loopback: Failed RTM_NEWADDR: Operation not permitted`). Поэтому промпт должен быть **самодостаточным** — весь контекст для ревью передаётся inline в теле промпта, без расчёта на то, что Codex откроет файлы.

Если первый запуск вернул сообщение вида «couldn't read files» / «sandbox error» / пустой результат — это всегда **sandbox fail, не технический refusal**. Перезапустить с diff'ом inline.

1. После commit, **перед push**, собрать полный контекст:
   ```bash
   git show --stat --oneline HEAD
   git show --find-renames --find-copies --format=fuller HEAD
   git status --short
   ```
   Для новых файлов и больших diff — включить релевантные фрагменты inline, если внешний reviewer не сможет прочитать файловую систему.

2. Вызвать ревью сторонней моделью с промптом, который **обязательно** содержит:
   - **PRD задачи** (полный текст из `/PRD/{stage}.md` или 5–10 строк описания цели, scope, ограничений)
   - **Ключевые факты про API-контракт** (имена полей, эндпоинты, enum'ы), проверенные по artifacts
   - **Committed diff** (`git show --find-renames --find-copies --format=fuller HEAD`) inline в fenced code block или доступный reviewer'у локально
   - **Содержимое новых файлов** (или ключевых их частей) inline, если reviewer не может читать файловую систему
   - **Test results** с количеством passed/failed
   - **Явные вопросы** для ревью — что проверять
   - Для sandboxed reviewer: **явное указание** «не трогай файловую систему, весь контекст уже в промпте»

   Шаблон промпта:
   ```
   Ревью сторонней моделью committed diff. Task: {краткое название}.
   If filesystem access is unavailable, use only the inline context.

   PRD (scope and intent):
   {содержимое PRD/{stage}.md или 5-10 строк описания цели,
    подзадач, ограничений}

   API contract facts (verified from project artifacts):
   - {endpoint 1}: filterable fields — {field names, types, operators}
   - {endpoint 2}: enum {name}: {values}
   - {важные связи, mapping внешних имён во внутренние}

   === COMMITTED DIFF ===
   ```diff
   {полный вывод `git show --find-renames --find-copies --format=fuller HEAD` здесь}
   ```

   === NEW FILE: {path} ===
   {содержимое нового файла целиком или ключевая часть}

   Test results: {N passed, M failed, suite command}

   === REQUEST ===
   Review for:
   1. {конкретный аспект 1}
   2. {конкретный аспект 2}
   3. {edge cases, о которых я переживаю}

   Format findings as: severity (critical/warning/nit), file:function, description.
   Keep under {300-500} words.
   ```

3. Если sandbox-ошибка случилась до чтения контекста — считать попытку не состоявшейся, перезапустить с inline-контекстом. Если reviewer стабильно недоступен, зафиксировать в AssumptionLog пропуск ревью сторонней моделью с обоснованием и указать это в self-attestation.

### 5.2 Оценка адекватности замечаний

Сторонняя модель может предлагать улучшения, которые **выходят за scope текущей задачи** или относятся к гипотетическим проблемам. Перед исправлением каждого замечания провести оценку:

| Критерий | Адекватно | Неадекватно |
|----------|-----------|-------------|
| **Scope** | Относится к коду в diff | Касается legacy кода или соседних модулей |
| **Реальность** | Указывает на конкретную ошибку или риск | Гипотетический "может быть в будущем" |
| **PRD** | Соответствует целям этапа | Расширяет задачу за пределы PRD |
| **ROI** | Fix маленький, риск регрессии низкий | Большой рефакторинг для nit |

**Адекватные** — исправлять обязательно (если critical) или по приоритету (warning).
**Неадекватные** — задокументировать в commit message или AssumptionLog с обоснованием отказа, не делать.

### 5.3 Severity reaction

- **critical** (адекватный) — обязательно исправить → тесты → аудит → commit → повторное ревью сторонней моделью (см. лимит в 5.4)
- **critical** (неадекватный) — задокументировать почему отклонено в AssumptionLog, продолжить
- **warning** (адекватный) — исправить если разумно
- **warning** (неадекватный) — пропустить с пояснением
- **nit** — на усмотрение, можно пропустить

### 5.4 Лимит итераций ревью

Максимум **2 запуска ревью сторонней моделью на code/diff review одной задачи** (1 первичный + 1 повторный). Это отдельный бюджет от PRD-review сторонней моделью.

Если после 2 итераций всё ещё остаются адекватные critical:
- **STOP**: не запускать ревью в 3-й раз
- Задокументировать оставшиеся findings в AssumptionLog с пометкой "deferred after 2 external-review iterations"
- Решить: либо вернуться к задаче в отдельном этапе, либо закоммитить с явным acknowledgement технического долга
- Спросить пользователя если непонятно как поступить

Цель лимита: избежать infinite loop, когда каждая итерация сторонней модели генерирует новые findings вместо подтверждения чистоты кода.

### 5.5 Когда можно пропустить ревью сторонней моделью

- Изменения только в документации (`.md` файлы, без кода)
- Изменения только в конфигах CI/CD без логики

## 5a. Deep Review — `/super-review` (по требованию)

Многоплановый ревью через skill `/super-review`. **Запускается только по явному запросу пользователя** — не в автоматическом цикле, не после каждого merged этапа.

**Когда запускать (только по явному «сделай ревью» / «проверь»):**
- Пользователь просит проверить накопленный за N этапов drift
- Перед релизом (если пользователь этого хочет)
- Для baseline audit при входе в новый крупный кусок работы
- Когда есть подозрение на systemic проблему, которая могла ускользнуть от per-commit Codex

**Агент НЕ должен:**
- Запускать `/super-review` сам после merge этапов
- Включать deep-review в Core Loop per-stage workflow (раздел 4) — там остаётся только ревью сторонней моделью на committed diff (раздел 5)
- Предлагать deep-review «на всякий случай» — это дорого (9 параллельных агентов + Codex), запускается осознанно

**Что запускается (в параллель):**
- 9 специализированных ревьюеров: `code`, `architecture`, `docs`, `security`, `performance-and-reliability`, `observability`, `tests`, `product`, `simplicity` + `codex-blindspot` (параллельный Codex pass) + `aggregator` = 11 subagent'ов в `.claude/agents/`
- `codex-blindspot`: один Codex-пробег с анти-корреляционным промптом (ищет то, что Claude-модели системно пропускают)
- `scripts/review_workflow_check.sh` — механические проверки (PRD файл, Roadmap статус, AssumptionLog, diff size)
- Агрегатор (`reviewer-aggregator`) — дедупликация, ранжирование, итоговый verdict
- Финальный Codex-арбитраж на top-10 blocker+high findings

**Scope (аргумент skill):**
- `changed` (дефолт) — `git diff` с main + uncommitted
- `related` — changed + 1 уровень импортов
- `full` — весь кодбейз
- `stage:N` — файлы из PRD/этап-N-*.md

**Codex-интеграция:**
- Default on (флаг `--no-codex` для отключения)
- Каждый субагент может эскалировать **до 2 findings** с confidence 0.4-0.7 в Codex для валидации
- Финальный арбитраж — один промпт с inline snippets top-10 (≤ 50k токенов)
- Fallback при sandbox fail: retry × 1, затем секция «codex skipped» в отчёте

**Отчёт:** `artifacts/review/{YYYY-MM-DD}-{scope}-stage-{N}.md`, коммитится с остальным кодом.

**Файлы инфраструктуры:**
- `.claude/agents/reviewer-*.md` — определения 10 агентов
- `.claude/commands/super-review.md` — skill
- `scripts/review_workflow_check.sh` — mechanical checks

## 9. Тесты и проверки

- Unit + mock e2e: `docker compose --profile test run --rm test`
- Real API (нужны `TEST_DOMAIN`, `TEST_API_KEY`): `docker compose --profile test run --rm -e TEST_DOMAIN=<домен> -e TEST_API_KEY=<ключ> test`
- CI: `.github/workflows/test.yml` (unit + mock); `test-real.yml` — ручной запуск

### 9.1 Изменения зависимостей → обязательный rebuild

Если в commit меняется любой из файлов:
- `Dockerfile` (особенно секции `pip install`, добавление/удаление upper bounds)
- `requirements.txt` / `pyproject.toml` (если появятся)
- `docker-compose.yml` (image config)

то **обязательно** перед прогоном тестов сделать:
```bash
docker compose --profile test build test
```

**Почему**: docker layer cache может содержать старую версию зависимости. Без rebuild тесты могут "проходить" на устаревшем образе, скрывая регрессию (например, upper bound может откатить production-версию пакета на старую с breaking changes).

**Кейс**: этап 58 добавил `fastmcp<3`, что молча откатило 3.1.1 → 2.14.6 при следующем rebuild. 26 тестов сломались, потому что fastmcp 2.14.x убрал public `call_tool` с FastMCP instance. Регрессия была скрыта 4 коммита, пока test image не пересобрали.

### 9.2 Pre-commit чеклист для dependency changes

Перед commit любого изменения зависимостей:
1. Узнать **текущую** установленную версию: `docker run --rm <prod_image> pip show <package>`
2. Убедиться что новый pin **включает** текущую версию (не откатывает)
3. `docker compose --profile test build test` — fresh rebuild
4. `docker compose --profile test run --rm test` — полный прогон
5. Только после этого — commit → ревью сторонней моделью → push
