# Технические требования: Vetmanager MCP Server

**Автор:** Manus AI
**Версия:** 1.0
**Дата:** 27 февраля 2026 г.

## 1. Введение

Этот документ содержит подробные технические требования и архитектурный дизайн для **Vetmanager MCP Server**. Он предназначен для команды разработчиков и описывает технологический стек, компоненты, паттерны реализации и лучшие практики, которым необходимо следовать в течение жизненного цикла проекта. Архитектура спроектирована так, чтобы быть надежной, безопасной и поддерживаемой, соответствуя стандартам как Vetmanager API, так и Model Context Protocol (MCP).

## 2. Технологический стек

Сервер будет разработан на Python с использованием современных библиотек для асинхронного программирования и взаимодействия с API.

| Компонент | Технология/Библиотека | Версия/Стандарт | Обоснование |
| :--- | :--- | :--- | :--- |
| **Язык** | Python | 3.11+ | Современный, надежный, с отличной поддержкой асинхронных операций и богатой экосистемой библиотек. |
| **MCP Фреймворк** | `mcp-sdk` (FastMCP) | 1.2.0+ | Официальный Python SDK для MCP, предоставляющий высокоуровневый, основанный на декораторах подход (`@mcp.tool`), который упрощает определение инструментов из функций Python, их аннотаций типов и docstrings. [1] |
| **HTTP Клиент** | `httpx` | последняя | Полнофункциональный, `async`-совместимый HTTP-клиент для Python, необходимый для взаимодействия с Vetmanager REST API. |
| **Управление окружением** | `uv` | последняя | Используется внутри Docker-образа для установки и запуска зависимостей; хост-машина не требует установленного Python toolchain. |
| **Контейнеризация** | Docker | последняя | Для упаковки приложения в виде легковесного, переносимого микросервиса, что упрощает развертывание и обеспечивает согласованность между средами. |

## 3. Архитектура и дизайн

Сервер будет спроектирован как stateless-микросервис, работающий как адаптер между клиентом MCP и Vetmanager REST API. Он будет следовать модульной, многоуровневой архитектуре.

### 3.0. Модель запуска (обязательная)

1. Сервер, тесты и CLI-команды запускаются только через `docker compose`.
2. Запуск Python на хосте (`python`, `pip`, `uv`, `mcp`) не является частью рабочего процесса проекта.
3. Все служебные команды (линт, тесты, миграции, e2e) выполняются внутри контейнера приложения.
4. Контейнер должен работать от пользователя с тем же UID/GID, что и на хосте, чтобы избежать проблем с правами при изменении файлов в bind mount.

### 3.1. Основные компоненты

1.  **MCP Server (`server.py`):** Основная точка входа в приложение. Он будет отвечать за:
    *   Инициализацию экземпляра `FastMCP`.
    *   Загрузку конфигурации (домен и API-ключ).
    *   Регистрацию всех инструментов API.
    *   Запуск основного цикла сервера, прослушивающего запросы через указанный транспорт (по умолчанию stdio).

2.  **API Клиент (`vetmanager_client.py`):** Выделенный класс-клиент для обработки всего взаимодействия с Vetmanager API. Этот компонент будет инкапсулировать логику для:
    *   **Динамического определения URL:** При инициализации он будет принимать `domain` и получать корректный базовый URL из `https://billing-api.vetmanager.cloud/host/{domain}`. Результат будет кэшироваться в переменной экземпляра для предотвращения избыточных запросов.
    *   **Аутентификации:** Он будет управлять заголовком `X-REST-API-KEY`, прикрепляя его к каждому исходящему запросу.
    *   **Методов запроса:** Он будет предоставлять общие, асинхронные вспомогательные методы для `GET`, `POST`, `PUT` и `DELETE` запросов, обрабатывая сериализацию запросов/ответов (JSON) и базовую проверку ошибок.
    *   **Обработки ошибок:** Он будет перехватывать исключения `httpx` (например, `HTTPStatusError`, `TimeoutException`) и преобразовывать их в стандартизированные исключения уровня приложения.

3.  **Определения инструментов (`tools/`):** Для поддержания порядка в кодовой базе, инструменты MCP будут определены в отдельных модулях Python в каталоге `tools/`. Каждый модуль будет соответствовать сущности Vetmanager API (например, `tools/client.py`, `tools/pet.py`).
    *   Каждая функция будет декорирована с помощью `@mcp.tool()`.
    *   Сигнатуры функций, аннотации типов и docstrings будут тщательно проработаны для создания четких, понятных для LLM спецификаций инструментов.
    *   Каждая функция-инструмент будет создавать экземпляр `VetmanagerAPIClient` и использовать его для выполнения необходимого вызова API.

### 3.3. Процесс аутентификации и конфигурации

1.  Сервер запускается и пытается загрузить `domain` и `api_key` из переменных окружения (`VETMANAGER_DOMAIN`, `VETMANAGER_API_KEY`).
2.  Если они не найдены, он корректно завершает работу с четким сообщением об ошибке, инструктирующим пользователя, как их установить.
3.  Основной файл `server.py` создает экземпляр `VetmanagerAPIClient` с этими учетными данными.
4.  Конструктор `VetmanagerAPIClient` немедленно вызывает эндпоинт для динамического определения URL и сохраняет базовый URL.
5.  Этот экземпляр клиента передается или импортируется функциями-инструментами для выполнения их операций.

## 4. Детали реализации

### 4.1. Генерация инструментов

Инструменты будут реализованы **статически**. Хотя спецификация OpenAPI может быть использована для динамической генерации, для начальной версии выбран статический, ручной подход, чтобы обеспечить высокое качество и человеко-читаемые docstrings и описания параметров, что критически важно для надежного взаимодействия с LLM.

**Пример инструмента (`tools/client.py`):**

```python
from mcp.server.fastmcp import FastMCP
from ..vetmanager_client import VetmanagerAPIClient

mcp = FastMCP("vetmanager") # Предполагается, что экземпляр mcp является общим

@mcp.tool()
async def get_client_by_id(client_id: int) -> dict:
    """Получает подробную информацию о конкретном клиенте по его уникальному ID.

    Args:
        client_id: Уникальный числовой ID клиента для получения.
    """
    api_client = VetmanagerAPIClient() # Упрощено для примера
    return await api_client.get(f"/rest/api/client/{client_id}")
```

### 4.2. Обработка ошибок

*   `VetmanagerAPIClient` будет вызывать кастомные исключения (например, `APIError`, `AuthenticationError`, `NotFoundError`).
*   Функции-инструменты будут содержать блоки `try...except` для перехвата этих конкретных исключений.
*   При перехвате исключения инструмент вернет удобное для пользователя, описательное сообщение об ошибке в виде строки, которое будет передано клиенту MCP.

### 4.3. Логирование

*   Будет использоваться стандартная библиотека Python `logging`.
*   Логирование будет настроено на запись в `stderr`, чтобы не повредить транспорт `stdio`, используемый MCP. [2]
*   Уровень логирования будет настраиваться через переменную окружения (например, `LOG_LEVEL=INFO`).
*   Логи будут включать временные метки, уровень логирования и идентификатор корреляции для отслеживания запросов в системе.

## 5. Стратегия тестирования

*   **Модульные тесты (Unit Tests):** `VetmanagerAPIClient` будет тщательно протестирован с помощью модульных тестов. `httpx.MockTransport` будет использоваться для симуляции ответов API и тестирования логики клиента для аутентификации, определения URL и обработки ошибок без реальных сетевых вызовов.
*   **Интеграционные тесты:** Будет разработан набор интеграционных тестов для каждого инструмента MCP. Эти тесты будут выполняться на живом (но в песочнице или тестовом) экземпляре Vetmanager. Они будут вызывать инструменты и проверять, что выполняются правильные вызовы API и что ответы форматируются должным образом.
*   **Ручное тестирование:** Сервер будет протестирован вручную с использованием клиента MCP, такого как MCP Inspector или совместимого чат-бота, для проверки сквозной функциональности и качества взаимодействия на естественном языке.
*   **Среда запуска тестов:** Все тесты запускаются только внутри контейнеров через `docker compose`, без зависимости от Python-окружения хоста.

## 6. Развертывание

*   Приложение будет упаковано в образ Docker с помощью предоставленного `Dockerfile`.
*   `Dockerfile` будет указывать на использование `uv` для установки зависимостей и устанавливать соответствующую точку входа (`uv run server.py`).
*   Пользователь будет отвечать за запуск контейнера Docker и предоставление необходимых переменных окружения (`VETMANAGER_DOMAIN`, `VETMANAGER_API_KEY`).
*   Основной способ запуска для локальной разработки и продакшн-подобных окружений — `docker compose`.
*   Конфигурация контейнера должна исключать проблемы прав доступа на хосте (запуск от non-root пользователя с UID/GID хоста и корректные права на volumes).
