## Executive Summary

Проект в рабочем состоянии: workflow-check зелёный, self-attestation stage 121 чистый. Но за последние 20+ этапов накопился заметный drift, который super-review проявил в трёх систеных темах:

1. **Контракт tool → VM API рассинхронизирован.** Минимум `tools/clinical.py`, `tools/finance.py`, `tools/client.py::create_client` отправляют camelCase payload на эндпоинты, которые ждут snake_case. VM API молча игнорирует неизвестные поля — `create_hospitalization(...)` создаёт hospital-запись с NULL `patient_id`/`doctor_id`, `create_payment(...)` — payment без client_id, `create_client(...)` — клиент без first_name/last_name. Это **уже в prod и приводит к silent data-loss**. Codex подтвердил и поднял severity c high до blocker.

2. **Тестовые фикстуры закрепляют неверный контракт.** `test_create_pet` постит `{client_id}` вместо `owner_id` (stage 77.4 мигрировал); `test_create_admission` постит `{pet_id, doctor_id, date}` вместо `{patient_id, user_id, admission_date}`. Эти тесты объясняют, почему #1 выжил 40+ этапов — они не ловят регрессию, а пинят её как expected.

3. **Reliability gaps сконцентрированы в auth/probe пути.** `vetmanager_connection_service.py` строит свежий `httpx.AsyncClient` на каждый вызов (нет pool reuse), делает single-attempt без retry, не эмитит `record_upstream_request`, вызывает `password.strip()` (RFC 8265 violation), и `_disable_existing_active_connections` без `SELECT FOR UPDATE` оставляет race на дубликат ACTIVE connection. Плюс `rate_limit_backend.RedisRateLimitBackend` использует sync-redis из async-handler'ов.

Остальное — предсказуемый drift: устаревшие doc-пути в README (4 broken references), §7.1 technical-requirements заканчивается на stage 116, «35 сущностей» vs actual 38, CLAUDE.md §5a говорит «10 subagents» вместо 11. Observability дыры — silent OPEN→HALF_OPEN breaker, отсутствие correlation_id в audit log, broken chain на non-FormPayloadTooLarge 500s.

## Verdict

**Do not merge as-is для stage 122+ release.** Текущий main не деградация, но перед следующим feature-этапом обязательны:

- **blocker**: fix camelCase payload в `clinical.py`/`finance.py`/`client.py::create_client` — это data-loss в prod
- **high**: переписать 2-3 broken test fixtures, иначе любой fix контракта не поймается CI
- **high**: async-Redis в rate_limit_backend (при первом Redis hiccup'е блокирует loop)
- **high**: устранить N+1 в `get_inactive_pets` + multi-clinic over-block в `get_doctor_free_slots`

Остальные findings медленно горят; фиксить в отдельных sweep-этапах.

## Systemic themes

1. **Field-name contract drift tools → VM API.** Minimum 5 tool modules send camelCase where snake_case is required; VM silently ignores unknown keys → data-loss at rest. The presence of test fixtures that encode the wrong contract (test_create_pet uses `client_id`; test_create_admission uses `pet_id`/`doctor_id`/`date`) is what prevented CI from catching this for 40+ stages. **Proposed remediation:** A contract-test harness that for every `create_*`/`update_*` tool asserts actual HTTP body field names against `api_entity_reference-ru.md`. This should be a standalone Roadmap stage (let's call it 122).

2. **Reliability concentration in auth-probe + rate-limit paths.** `vetmanager_connection_service.py` accumulates 5 orthogonal issues (pool, retry, metrics, strip, race); `rate_limit_backend.py` uses sync-redis from async; `runtime_auth.py` does 5+ DB roundtrips per client init. These aren't random — they're all "every request goes through this" hot path. One coordinated reliability sweep yields disproportionate SLO gains.

3. **BC-shim soup / incomplete migrations.** 6 top-level shim modules + 25 aliased re-exports in `vetmanager_client.py` with near-zero non-test consumers. Each was transitional for stages 91/103/106; per AssumptionLog stage 114b the policy was "KEEP-all-shims". Simplicity eval dispute: at minimum `ttl_for_entity()` and `build_list_query_params` re-export have 0 call-sites anywhere. Proposed: one consolidation stage deleting dead shims (stages 122 or 123).

4. **Observability sparse in silent-failure branches.** Breaker OPEN→HALF_OPEN is silent; `evaluate_connection_health` swallows its own errors; `_observed_custom_route` silent on non-FormPayloadTooLarge 500s; auth-probes don't emit `record_upstream_request`. Pattern: where we most need signal, logging is missing.
