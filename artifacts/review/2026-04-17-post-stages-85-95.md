### 1. `update_admission` не починен — та же проблема что F1 в `create_admission`
- severity: **blocker** (product), confidence: 0.95
- file: `tools/admission.py:335-378`
- problem: Stage 86 исправил `create_admission` (pet_id→patient_id, doctor_id→user_id, date→admission_date). `update_admission` продолжает строить payload с broken именами. Vetmanager молча игнорирует неизвестные поля → reschedule приёма / reassign врача / reassign пета — noop.

- confirmed_by: [product]

### 2. `get_client_profile` использует phantom enum `status='active'`
- severity: **high** (product), confidence: 0.9
- file: `tools/client.py:381-387`
- problem: Filter `next_admission` строится с `{"property": "status", "value": "active"}`. Такого значения в admission.status enum нет (см. api-research-notes-ru.md: save/directed/accepted/deleted/delayed/not_approved/in_treatment/not_confirmed). Фильтр матчит 0 строк → `next_admission` всегда `None`.
- why_it_matters: Docstring обещает «следующий запланированный приём», возвращает `None` на 100% реальных инсталляций. Персона Мария («какой следующий визит у клиента X?») получает пусто. Стадия 95 трогала этот файл (добавила partial-response), но сам баг не заметила.
- suggested_fix: Заменить на `operator: "IN"` с tuple'ом `ACTIVE_ADMISSION_STATUSES` (уже определён в `tools/admission.py`, вынести в shared constants). Регресс-тест на URL-фильтр.
- confirmed_by: [product]

### 3. Partial-gather глотает `asyncio.CancelledError`
- severity: **high** (semantics bug — codex-blindspot), confidence: 0.98
- file: `tools/client.py:395-419`
- problem: `asyncio.gather(..., return_exceptions=True)` возвращает CancelledError в результатах. `isinstance(resp, BaseException)` ловит CancelledError (она — BaseException, не Exception). Cancellation конвертируется в `partial:true` вместо propagation.
- why_it_matters: Disconnected MCP клиент или shutdown не останавливает дорогие upstream задачи; нижележащий код продолжает обрабатывать результат, возвращая stale partial вместо честного прерывания.
- suggested_fix: Перед `isinstance(..., BaseException)` проверить `isinstance(resp, asyncio.CancelledError): raise`. Или использовать `isinstance(resp, Exception)` (не BaseException) — стандартный паттерн для return_exceptions.
- confirmed_by: [codex-blindspot, performance-and-reliability]

### 4. Circuit breaker HALF_OPEN: probe_in_flight не очищается на 4xx
- severity: **high** (reliability), confidence: 0.92
- file: `vetmanager_client.py:154-188, 515-518`
- problem: В HALF_OPEN `_check_breaker_allows` ставит `probe_in_flight=True`. Если probe возвращает 404/401/403 (AuthError/NotFoundError/VetmanagerError) — код идёт в ветку `except (AuthError, NotFoundError, VetmanagerError): raise` БЕЗ вызова `_breaker_record_success` или `_breaker_record_failure`. `probe_in_flight` остаётся True навсегда → все последующие callers fast-fail с `circuit_half_open_busy`.
- why_it_matters: После транзиентного инцидента один 404 на probe-запросе (например `get_doctor_free_slots` для удалённого доктора) залипает домен в half_open до рестарта процесса.
- suggested_fix: В `except (AuthError, NotFoundError, VetmanagerError)` ветке вызвать `_breaker_record_success(domain_key)` — 4xx означает upstream жив (хотя запрос не прошёл). Regression test: probe → 404 → next caller должен пройти.
- confirmed_by: [performance-and-reliability, architecture, codex-blindspot]

### 7. Timeout/network/retry warnings без correlation_id
- severity: **high** (observability), confidence: 0.95
- file: `vetmanager_client.py:502-513, 531-542, 450-463`
- problem: Structured warnings `vm_upstream_timeout`, `vm_upstream_network_error`, `vm_upstream_retry` не включают `correlation_id` в extra dict. При concurrent requests к одному domain несколько timeouts с одинаковым `(domain, method, url_path)` невозможно разделить.
- why_it_matters: Триаж на `get_client_profile` timed out at 14:23 — нужно correlate с incoming MCP request, но без correlation_id это только сопоставление по timestamp (хрупко при concurrency).
- suggested_fix: Capture `correlation_id` из `get_current_request_context()` до retry loop (один раз), включить в extra dict во всех 3 warning logs.
- confirmed_by: [observability]

### 8. `circuit_open` в отдельной метрической семье от upstream_requests_total
- severity: **high** (observability), confidence: 0.90
- file: `service_metrics.py:71-74` vs `vetmanager_client.py:175`
- problem: `record_upstream_failure(reason="circuit_open")` пишет в `_UPSTREAM_FAILURES_TOTAL`; реальные HTTP calls пишут через `record_upstream_request` в `_UPSTREAM_REQUESTS_TOTAL + _UPSTREAM_LATENCY_SECONDS`. Две разные prom-families без общих labels.
- why_it_matters: Оператор считающий «upstream error rate» = failures/total через единый counter недосчитает circuit-open fast-fails → дашборды lie именно во время инцидента.
- suggested_fix: Также вызывать `record_upstream_request(target="vetmanager_api", status="circuit_open", duration_seconds=0.0)` при circuit fast-fail. Единая counter family.
- confirmed_by: [observability]

### 9. `get_client_profile` не инструментирован (не видно в tool_calls_total/latency)
- severity: **high** (observability), confidence: 0.93
- file: `tools/client.py:357-440`
- problem: Aggregator делает 4 parallel upstream calls через `vc.get()` напрямую, минуя `_instrumented_call`. В `vetmanager_tool_calls_total` и `tool_call_latency_seconds` нет записи для самого high-fanout тула.
- why_it_matters: Dashboards per-tool latency слепы к главному aggregator'у. Capacity planning и SLO tracking для get_client_profile невозможны через текущую телеметрию.
- suggested_fix: Обернуть тело `get_client_profile` в `_instrumented_call("/rest/api/client", "GET", _coro_factory)`. То же для `get_pet_profile` и `paginate_all` callers.
- confirmed_by: [observability, architecture]

### 10. Breaker failures недосчитываются внутри retry-loop
- severity: **medium**, confidence: 0.70
- file: `vetmanager_client.py:247-255, 490-494`
- problem: При 3x retry на 5xx в одном tool call `_breaker_record_failure` вызывается 1 раз (в finally exhausted retries), но реально было 3 upstream failures. Threshold 5 → нужно 5 tool calls (15 upstream attempts) чтобы breaker открылся.
- why_it_matters: Breaker реагирует медленнее intended. При genuinely down upstream 1 tool caller не откроет breaker — потребуется 5x retry exhausted.
- suggested_fix: Записывать `_breaker_record_failure` на каждом retryable 5xx/timeout attempt, не только на terminal. Или считать exhausted attempts как N failures.
- confirmed_by: [performance-and-reliability, codex-blindspot]

## Systemic themes

1. **Sweep discipline gap after field-mapping fix (stage 86).** `create_admission` исправлен, `update_admission` — нет. `get_client_profile::status='active'` — phantom enum никто не grep'ал. Нужен lint/CI check: grep всех `{"property": "<admission field>"` и проверять против `ACTIVE_ADMISSION_STATUSES` enum, и grep payload keys против authoritative entity dict. Это то, что стадия 87 обещала в 87.3 (CI lint), но отложила в `stop`.

2. **Half-adopted abstractions.** FilterBuilder ship'нут без миграции, partial-gather ship'нут только в `get_client_profile` (не в `get_pet_profile`), Sentry sanitizer покрывает только request.headers/body/etc но не breadcrumbs/stacktrace vars, observability обёрнута только у crud_*, но не у aggregators. Паттерн: каждая стадия вводит механизм, следующая должна была бы мигрировать callers, но это уходит в X-b deferred работу. Дебт копится.

3. **Observability blind spots в собственной observability-инфраструктуре.** correlation_id в warnings нет, aggregators без tool_calls_total, circuit_open в отдельной metric family, 4xx считается как upstream_failure. Стадия 88 добавила правильные поля/метрики, но callers не пользуются. Runbook операционной диагностики надо перепроверить на основе актуального metric set'а.

4. **Deferred-work accumulation без дедлайнов.** 5 consecutive stages (91-95) с паттерном «малую часть сделали, остальное deferred». Сам по себе не проблема, но ни один из 92b/93b/94b/95b не имеет owner или ETA. Через ещё 2-3 таких цикла будет «вечный tech debt».

5. **Тесты coupled к internals.** `_shared_http_client`, `_breakers[DOMAIN]`, `breaker.opened_at`, `PROMPTS_SRC.index(...)` — все это breaks при refactor'е. Нужен публичный test-API layer: `get_shared_http_client()`, `get_breaker_state(domain)`, prompt helpers. Stage 94b как раз про это — приоритизировать.
