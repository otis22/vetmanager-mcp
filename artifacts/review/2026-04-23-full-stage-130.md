## Executive Summary

Full review после stage 130 нашёл один blocker и несколько high-risk проблем вокруг нового depersonalized bearer runtime. Главный риск: деперсонализация сейчас применена как post-filter после tool execution и при повторной ошибке auth lookup возвращает raw payload. Второй кластер — неполная privacy schema: реальные VM поля `home_phone`, `work_phone`, `diagnos`, `recomendation`, `note` и похожие поля не покрыты sanitizer whitelist. Третий кластер — scope/preset рассинхрон: `ClientPhone` не защищён `clients.read`, `frontdesk` не может вызвать slots tool, а `TOOL_REQUIRED_SCOPES` пока не является runtime preflight. Дополнительно подтверждены drift в datetime contract tests, lifecycle cleanup Redis rate-limit backend, audit/correlation gaps и stale technical requirements.

## Verdict

**Do not merge as release candidate** — stage 130 функционально закрыт, но privacy boundary для depersonalized токенов имеет blocker fail-open path и неполное покрытие PII-полей. Исправления должны идти первым новым этапом перед дальнейшим rollout.

## Top Findings

### B1. Depersonalization wrapper can fail open

- severity: blocker
- confirmed_by: architecture, codex-blindspot, product, security, simplicity, Claude arbitration
- file: `tools/__init__.py:12-26`
- problem: wrapper runs the tool first, then calls `resolve_runtime_credentials()` again; `AuthError` returns raw result.
- fix: resolve bearer context once before tool execution, share it with `VetmanagerClient` and sanitizer, fail closed when depersonalization state cannot be determined.

### H1. Structured phone redaction misses real VM fields

- severity: high
- confirmed_by: code, codex-blindspot, security, spark-file, Claude arbitration
- file: `depersonalization.py:26`, `tools/client.py:178-179`, `tools/pet.py:278`
- problem: `_PHONE_KEYS` covers only `phone`/`cellphone`; `home_phone`, `work_phone`, `owner_phone` can leak.
- fix: expand normalized phone key coverage and regression-test depersonalized `get_debtors`, client payloads and inactive pets owner payload.

### H2. Free-text sanitizer misses VM-native clinical/note fields

- severity: high
- confirmed_by: code, codex-blindspot, Claude arbitration
- file: `depersonalization.py:29`, `tools/medical_card.py:199-204`, `tools/pet.py:105-154`
- problem: whitelist covers English `diagnosis`/`notes`, but VM uses `diagnos`, `recomendation`, `note`, and related fields.
- fix: derive/extend curated free-text key list from OpenAPI/API reference and add corpus tests.

### H3. Admission/hospital datetime tests codify wrong VM format

- severity: high
- confirmed_by: codex-blindspot, Claude arbitration
- file: `tools/admission.py:278-289`, `tools/clinical.py:44-58`, `tests/test_api_contracts_hotfix.py`
- problem: tools pass ISO `T` datetimes unchanged while VM contract requires `Y-m-d H:i:s`.
- fix: normalize accepted datetime input to space-separated VM format and update contract tests; run real smoke if credentials are available.

### H4. `ClientPhone` endpoint lacks `clients.read` scope

- severity: high
- confirmed_by: codex-blindspot, security, Claude arbitration
- file: `token_scopes.py:52-88`, `tools/client.py:112-130`
- problem: `required_scope_for_request()` returns `None` for `/rest/api/ClientPhone`, allowing phone-index probing without `clients.read`.
- fix: map `clientphone` to `clients.read` and add denied-token phone-search regression.

### H5. `frontdesk` preset cannot use slots despite PRD intent

- severity: high
- confirmed_by: docs, product, spark-file, Claude arbitration
- file: `tool_access_registry.py:67-80,164`, `PRD/этап-130-token-presets-and-depersonalized-bearer-tokens.md:21-26`
- problem: `frontdesk` lacks `analytics.read`, while `get_doctor_free_slots` requires it.
- fix: add `analytics.read` to frontdesk or narrow product contract/UI; preferred fix is add read-only scope and preset/tool matrix tests.

## Medium Findings

### M1. Tool registry is not runtime preflight

- file: `tool_access_registry.py:116-232`, `vetmanager_client.py:245-257`
- issue: `TOOL_REQUIRED_SCOPES` is validated by tests but runtime enforcement still derives from REST path. Aggregate tools can partially execute before a later upstream call fails.
- roadmap: stage 132.

### M2. Redis rate-limit backend lifecycle is not closed by server shutdown

- file: `server.py:48-72`, `rate_limit_backend.py`
- issue: stage 129 added explicit backend close, but `_graceful_shutdown()` does not call it.
- roadmap: stage 134.

### M3. Audit log is emitted before durable commit and lacks correlation ids

- file: `auth_audit.py:97-122`
- issue: `AUDIT_LOGGER.info("Recorded token audit event.")` runs before caller commit; audit details/logs do not carry request/correlation ids.
- roadmap: stage 134.

### M4. Web error/correlation observability gaps

- file: `web.py:182-198`, `web_routes_system.py:57-64`, `server.py:106-110`, `web_routes_account.py:290-305`
- issue: generic custom-route 500s lack structured logs, 413 response bypasses correlation headers, unauthorized `/metrics` has no security signal, startup abort uses root logger, token lifecycle logs lack explicit correlation id.
- roadmap: stage 134.

### M5. Host resolver cold-cache calls are not coalesced

- file: `host_resolver.py:134-147`
- issue: concurrent cold-cache calls for one domain can all hit billing API before cache is populated.
- roadmap: stage 134.

### M6. Additional API/tool correctness gaps

- issue: `get_timesheets(date=...)` uses containment rather than overlap semantics, excluding overnight shifts; `get_message_reports(campaign=\"\")` silently drops campaign via query builder.
- roadmap: stage 133.

### M7. Stage 130 test gaps

- files: `tests/test_stage130_access_registry.py`, `tests/test_token_scopes.py`, `tests/test_web_auth.py`, `tests/test_prometheus_metrics.py`, `tests/test_stage130_depersonalization.py`
