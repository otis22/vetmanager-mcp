# Требования к продукту (PRD): Vetmanager MCP Server

**Автор:** Manus AI
**Версия:** 1.3
**Дата:** 23 марта 2026 г.

## 1. Введение

Этот документ определяет требования к продукту **Vetmanager MCP Server** — специализированному сервису, предназначенному для интеграции Vetmanager REST API с любым клиентским приложением, поддерживающим Model Context Protocol (MCP). Основная цель — обеспечить бесшовное взаимодействие с платформой Vetmanager на естественном языке, позволяя пользователям управлять операциями ветеринарной клиники через диалоговых AI-агентов, чат-ботов и другие MCP-совместимые инструменты.

Этот сервер будет действовать как стандартизированная интеллектуальная «обертка» для Vetmanager API, переводя команды на естественном языке в конкретные вызовы API и возвращая структурированные, удобочитаемые результаты. Он возьмет на себя сложности аутентификации, динамического определения URL-адресов API и форматирования данных, предоставляя простой, но мощный интерфейс как для конечных пользователей, так и для разработчиков.

## 2. Видение и цели продукта

### 2.1. Видение

Наше видение — предоставить сотрудникам ветеринарной клиники, от администраторов до врачей, возможность взаимодействовать с системой управления клиникой так же легко, как если бы они разговаривали с ассистентом. Абстрагируясь от сложностей программного обеспечения, мы стремимся сократить административную нагрузку, минимизировать время на обучение и позволить специалистам сосредоточиться на лечении пациентов.

### 2.2. Цели

*   **Упрощение операций:** Позволить пользователям выполнять рутинные задачи, такие как планирование приемов, выставление счетов и ведение записей, с помощью простых диалоговых команд.
*   **Повышение доступности:** Предоставить стандартизированный, удобный для разработчиков интерфейс к Vetmanager API, способствуя развитию экосистемы сторонних интеграций и кастомных решений.
*   **Повышение эффективности:** Автоматизировать повторяющиеся задачи по извлечению и вводу данных, экономя время и снижая вероятность человеческой ошибки.
*   **Обеспечение безопасности:** Предоставить безопасный механизм для обработки учетных данных API и управления доступом к конфиденциальным данным клиники и пациентов.

## 4. Функциональные требования

### 4.1. Основная функциональность: Реализация инструментов MCP

Сервер должен предоставлять всю широту возможностей Vetmanager API в виде набора инструментов MCP. Это включает в себя сопоставление эндпоинтов API с логическими, обнаруживаемыми функциями, которые AI-модель может понять и вызвать.

*   **Требование 4.1.1:** Все основные сущности из Vetmanager API должны быть представлены в виде инструментов. Это включает, но не ограничивается: `Client` (клиент), `Pet` (питомец), `Admission` (прием), `MedicalCard` (медкарта), `Invoice` (счет), `Good` (товар/услуга) и `User` (пользователь).
*   **Требование 4.1.2:** Инструменты должны поддерживать полные CRUD-операции (Create, Read, Update, Delete), где это позволяет базовый API.
*   **Требование 4.1.3:** Каждый инструмент должен иметь четкое, описательное имя и исчерпывающую документацию (docstring), объясняющую его назначение, параметры и возвращаемое значение, что обеспечивает точный вызов со стороны LLM.
*   **Требование 4.1.4:** Инструменты должны корректно обрабатывать параметры API для фильтрации, сортировки и пагинации (`limit`, `offset`, `sort`, `filter`) во всех list GET методах.
*   **Требование 4.1.5:** Параметры `sort` и `filter` должны поддерживать формат Vetmanager API и передаваться как JSON-строки в query-параметрах. Для `filter` должны поддерживаться операторы: `=`, `!=`, `<>`, `<`, `<=`, `>`, `>=`, `in`, `not in`, `like`.
*   **Требование 4.1.6:** Сервер должен поддерживать не только стандартные CRUD-инструменты, но и специальные доменные операции Vetmanager API, которые не укладываются в обычную схему сущность-by-id. К ним относятся агрегирующие профили клиента и питомца, история вакцинаций и глобальные уведомления `messages/*`.
*   **Требование 4.1.7:** Сервер должен публиковать типовые MCP prompts для распространённых сценариев администратора, врача, финансового блока и склада, чтобы LLM-клиенты могли использовать готовые workflow поверх набора инструментов.
*   **Требование 4.1.8:** `tools/list` должен быть источником истины по возможностям сервера и возвращать для каждого инструмента как минимум `name`, `description` и `inputSchema`, пригодные для автоматического выбора инструмента со стороны MCP-клиента.
*   **Требование 4.1.9:** Описания инструментов в `tools/list` должны формироваться из актуальных docstrings и доменных синонимов сущностей, чтобы LLM корректно сопоставляла пользовательские формулировки вроде «хозяин», «запись на приём», «остаток на складе» или «прививка» с нужным инструментом.
*   **Требование 4.1.10:** Для параметров с жёсткими safety-ограничениями, прежде всего `limit`, сервер должен публиковать ограничения прямо в `inputSchema` (`minimum`, `maximum`, описание), чтобы MCP-клиенты и LLM не генерировали заведомо невалидные вызовы.

### 4.2. Аутентификация и конфигурация

Сервер должен управлять как внешней аутентификацией в самом сервисе, так и
внутренней авторизацией к Vetmanager API от имени аккаунта сервиса.

*   **Требование 4.2.1:** Пользователь должен проходить регистрацию и работать с сервисом через аккаунт, в котором хранится конфигурация подключения к Vetmanager.
*   **Требование 4.2.2:** MCP-клиенты должны аутентифицироваться в сервисе только через `Authorization: Bearer <service_token>`.
*   **Требование 4.2.3:** Bearer-токены принадлежат аккаунту сервиса; один аккаунт может выпускать несколько Bearer-токенов с независимыми сроками действия и статусами.
*   **Требование 4.2.4:** В аккаунте должен быть настроен ровно один активный способ авторизации в Vetmanager, который используется всеми Bearer-токенами этого аккаунта.
*   **Требование 4.2.5:** Сервис должен поддерживать два способа авторизации в Vetmanager:
    `domain + rest_api_key` и `user login/password -> user token`.
*   **Требование 4.2.6:** Сервис должен сохранять поддержку обоих Vetmanager auth modes, при этом `domain + rest_api_key` остаётся более простым базовым способом подключения, а `user login/password -> user token` поддерживается как отдельный web-driven flow.
*   **Требование 4.2.7:** При использовании mode `domain + rest_api_key` сервер должен выполнить GET-запрос к `https://billing-api.vetmanager.cloud/host/{domain}` для получения динамического базового URL клиники и использовать `X-REST-API-KEY` для последующих вызовов API.
*   **Требование 4.2.8:** Runtime credentials Vetmanager не должны передаваться как аргументы MCP-инструментов; сигнатуры инструментов содержат только бизнес-параметры.
*   **Требование 4.2.9:** Bearer-only модель является текущим runtime-контрактом сервиса; dual-mode поддержки не предусматривается.
*   **Требование 4.2.10:** При отсутствии, просрочке, отзыве или некорректности Bearer-токена сервис должен возвращать явную и безопасную ошибку без раскрытия секретов.
*   **Требование 4.2.11:** Значения `domain`, `rest_api_key`, логин, пароль и пользовательские токены Vetmanager должны храниться и обрабатываться как секреты и не раскрываться в логах, ошибках и артефактах проекта.
*   **Требование 4.2.12:** Значения, используемые в тестах, должны оставаться тестовыми данными и не должны быть захардкожены в коде приложения.
*   **Требование 4.2.13:** Сервис должен предоставлять web-контур аккаунта (`landing`, `register`, `login`, `account`) для настройки Vetmanager-интеграции и выпуска Bearer-токенов без ручного редактирования runtime-конфига сервера.
*   **Требование 4.2.14:** После создания Bearer-токена сервис должен показывать его сырой вид только один раз; далее в storage и UI допускаются только безопасные метаданные токена.
*   **Требование 4.2.15:** Для режима `user login/password -> user token` сервис должен выполнять token exchange через `POST /token_auth.php`, не сохранять логин/пароль и использовать полученный `user_token` по отдельному runtime-контракту Vetmanager.

### 4.3. Обработка ошибок и надежность

Сервер должен быть отказоустойчивым и предоставлять четкую обратную связь при возникновении проблем.

*   **Требование 4.3.1:** Сервер должен корректно обрабатывать и сообщать об ошибках от Vetmanager API, таких как коды состояния `4xx` (например, неверный ввод, не найдено) и `5xx` (ошибки на стороне сервера).
*   **Требование 4.3.2:** Сообщения об ошибках, возвращаемые пользователю, должны быть как удобочитаемыми, так и машинно-читаемыми, четко объясняя суть проблемы (например, «Предоставлен неверный API-ключ», «Клиент с ID 123 не найден»).
*   **Требование 4.3.3:** Сервер должен обрабатывать проблемы, связанные с сетью, такие как тайм-ауты при подключении к Vetmanager API, и сообщать о них соответствующим образом.
*   **Требование 4.3.4:** Сервер должен применять тегированное кеширование для всех успешных GET-запросов Vetmanager API: TTL 15 минут, ключ `method + canonical_full_url_with_query + api_key_hash`, инвалидация по тегу `domain:entity` при успешных `POST`/`PUT`/`DELETE` на соответствующую сущность.
*   **Требование 4.3.5:** Сервер должен применять safety-валидацию к массовым операциям и потенциально опасным параметрам: безопасные лимиты на list-запросы, ограничение суммы платежа и явные ошибки при нарушении этих ограничений.
*   **Требование 4.3.6:** Сервис должен вести для Bearer-токенов как минимум `last_used_at` и `request_count`, чтобы пользователь аккаунта видел последнее использование и объём трафика по токену.
*   **Требование 4.3.7:** Сервис должен поддерживать отзыв Bearer-токенов и корректно прекращать доступ по ним сразу после revoke.
*   **Требование 4.3.8:** Сервис должен предоставлять health/readiness/metrics контур и структурированные логи, достаточные для production-эксплуатации и расследования инцидентов. Реализованные эндпоинты: `/healthz` (liveness), `/readyz` (readiness с проверкой БД), `/metrics` (Prometheus-совместимые метрики).
*   **Требование 4.3.9:** Сервис должен применять rate limiting для защиты от brute-force и злоупотреблений: per-token rate limiting для Bearer runtime (1000 req/60s по умолчанию), per-IP rate limiting для web endpoints, per-email lockout для login (10 попыток/15 минут) и registration (3 попытки/1 час).
*   **Требование 4.3.10:** Сервис должен автоматически создавать backup БД перед каждым деплоем и проверять целостность данных после рестарта.

## 5. Нефункциональные требования

*   **Производительность:** Сервер должен вносить минимальную задержку. Поиск динамического URL должен кэшироваться, чтобы избежать повторных вызовов к API биллинга. Между последовательными исходящими HTTP-запросами к Vetmanager API внутри одного клиентского контекста должна соблюдаться минимальная пауза 50ms.
*   **Кеширование:** Для GET-запросов используется in-memory кеш в процессе сервера (без внешнего Redis/БД). Кеш должен учитывать query-параметры и API-ключ (через hash), а также поддерживать теговую инвалидацию по домену и сущности.
*   **Безопасность:** API-ключи и другие конфиденциальные данные должны безопасно обрабатываться в памяти и никогда не должны раскрываться в логах или сообщениях об ошибках. `domain` должен проходить строгую валидацию формата subdomain. Резолвленный host должен использовать HTTPS и проходить allowlist-проверку доменной зоны Vetmanager.
*   **Масштабируемость:** Сервер должен быть спроектирован как stateless-приложение, что позволяет горизонтально масштабировать его при развертывании в среде с высоким трафиком.
*   **Поддерживаемость:** Код должен быть хорошо структурирован, прокомментирован и сопровождаться юнит-тестами для облегчения будущих обновлений и обслуживания.
*   **Расширяемость:** Архитектура должна позволять в будущем ввести scopes / RBAC для Bearer-токенов без полной переработки auth-модели аккаунта.
